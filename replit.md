# CareConnect - Elderly Care Service Management System

## Overview
CareConnect is a full-stack, mobile-first web application designed to streamline operations for caregivers managing elderly care services. It provides functionalities for scheduling, tracking, and documenting appointments, including digital signatures and real-time data management. The system aims to enhance efficiency for care professionals, offer comprehensive customer management, unify service models, ensure robust data historization for compliance, and adhere to German labor laws. The business vision includes expanding customer management capabilities, unifying service models, and ensuring data integrity for auditing.

## User Preferences
- Preferred communication style: Simple, everyday language
- **Keine Avatare/Profilbilder**: Für Kunden und Mitarbeiter werden keine Fotos oder Avatar-Platzhalter verwendet. Stattdessen werden Namen direkt mit Badges (z.B. Pflegegrad) dargestellt. Dies spart Platz und hält die Oberfläche aufgeräumt.
- **Keine Blur-Effekte**: Kein `backdrop-blur`, kein `bg-black/80` oder ähnlich starke Overlay-Verdunkelung. Dialog-/Sheet-/Drawer-Overlays verwenden maximal `bg-black/50` ohne Blur-Filter. Die UI soll klar und technisch scharf bleiben.
- **Keine CSS-Transforms in Overlay-Komponenten**: Dialog, AlertDialog, Sheet und Drawer dürfen KEINE `translate`, `scale`, `zoom` oder `slide` CSS-Transforms verwenden. Diese verursachen Sub-Pixel-Rendering und unscharfen Text. Stattdessen: Flexbox-Zentrierung (`fixed inset-0 flex items-center justify-center`) und reine Fade-Animationen (`fade-in-0`/`fade-out-0`, nur opacity). Drawer: `shouldScaleBackground = false`. Ausnahme: Sheet-Slide-Animationen (`slide-in-from-*`/`slide-out-to-*`) sind erlaubt, da Sheets am Bildschirmrand positioniert sind und keine Sub-Pixel-Probleme verursachen.
- **Standard-Unterschrift-Komponente**: Für ALLE Unterschriften im System MUSS die zentrale `SignaturePad`-Komponente (`@/components/ui/signature-pad.tsx`) verwendet werden. KEINE eigenen Signature-Dialoge, Canvas-Implementierungen oder alternative Unterschriftenlösungen bauen. `SignaturePad` bietet eine konsistente Fullscreen-Unterschriftserfahrung mit „Tippen zum Unterschreiben"-Platzhalter, X-Markierung und einheitlichem Styling. Wird verwendet in: Kundenanlage (signatures-step), Leistungsnachweis-Unterschrift, digitaler Dokumentenfluss.

## System Architecture

### Frontend
- **Frameworks**: React 19 with TypeScript, Vite, Wouter for routing.
- **UI/UX**: Mobile-first responsive design using `shadcn/ui` components on Radix UI primitives, styled with Tailwind CSS v4 and a "Care & Clarity" theme. Centralized `@/design-system` for consistent styling. Touch-optimized UI components. Layouts use a warm beige gradient background. Reusable `Skeleton` and `PageSkeleton` components (`@/components/ui/skeleton.tsx`) for content-shaped loading states (variants: `default`, `detail`, `list`). All `<img>` tags MUST include `width` and `height` attributes for CLS prevention.
- **State Management**: TanStack Query for data fetching. `staleTime: 30s`, `refetchOnWindowFocus: true`. Centralized query invalidation system (`@/lib/query-invalidation.ts`): `invalidateRelated(queryClient, ...domains)` maps domain dependencies (e.g., appointment mutations auto-invalidate time-entries, service-records, budget, customers, notifications, auth). All mutations MUST use `invalidateRelated()` instead of manual `queryClient.invalidateQueries()` for cross-domain consistency. Navigation prefetching via `queryClient.prefetchQuery()` on hover/focus (AppointmentCard, CustomerCard) for smoother perceived navigation.
- **Date/Time Handling**: German local time, no UTC conversion.
- **API Calls**: Central API client (`@/lib/api/client.ts`) for all HTTP requests, including CSRF protection, with `unwrapResult()` for data extraction. Direct `fetch()` calls are forbidden.
- **Feedback**: German toast notifications on success and error for all `useMutation` hooks.
- **Phone Number Handling** (zentral in `@shared/utils/phone`):
  - **Speicherung**: Immer E.164 (`+4917632238795`) via `normalizePhone()`.
  - **Anzeige**: Immer `formatPhoneForDisplay()` verwenden — zeigt deutsches Nationalformat (z.B. `0176 32238795`) oder internationales Format für Nicht-DE-Nummern. Niemals rohe E.164-Werte im UI anzeigen.
  - **Eingabefelder**: `formatPhoneAsYouType()` für Live-Formatierung während der Eingabe.
  - **Validierung**: `validateGermanPhone()` vor dem Speichern aufrufen.
  - **tel:-Links**: `href="tel:${rawE164}"` mit dem rohen E.164-Wert, aber Anzeigetext immer via `formatPhoneForDisplay()`.
- **Adressformatierung** (zentral in `@shared/utils/format`):
  - **Anzeige**: Immer `formatAddress()` verwenden. Akzeptiert Objekte mit `strasse`, `nr`/`hausnummer`, `plz`, `stadt` und fällt auf das `address`-Feld zurück. Ausgabe: `"Strasse Nr, PLZ Stadt"`.
  - Niemals Adresskomponenten manuell im JSX zusammensetzen.
- **Km-Formatierung** (zentral in `@shared/utils/format`):
  - **Anzeige**: Immer `formatKm()` verwenden — gibt `"1,5"` zurück (deutsches Komma-Format, 1 Dezimalstelle). Re-Export in `@/lib/utils` für Frontend-Kompatibilität.
- **Logging** (zentral in `server/index.ts`):
  - **Alle Server-Logs**: Immer die zentrale `log(message, source)` Funktion verwenden. `console.log` ist nur innerhalb der `log`-Funktion selbst erlaubt. `console.error` für echte Fehler ist weiterhin OK.
  - **Format**: `"HH:MM:SS AM/PM [source] message"` — z.B. `5:12:54 PM [startup] Pflegekassen-Import: ...`
  - **Import**: `import { log } from "../lib/log"` (oder relativer Pfad je nach Tiefe). Re-Export in `server/index.ts` für Abwärtskompatibilität.

### Backend
- **Framework**: Express.js with TypeScript.
- **API Design**: RESTful endpoints, Zod validation, structured error responses, modular routing. Large route files are split into sub-routers (e.g., `server/routes/admin/customers/` with `assignments.ts`, `budgets.ts`, `contracts.ts`, `details.ts`, `workflows.ts`; `server/routes/statistics/` with `revenue.ts`, `operations.ts`, `customers.ts`; `server/routes/customers/` with `contacts.ts`, `service-prices.ts`, `documents.ts`; `server/routes/admin/employees.ts` → `employee-users.ts` (user CRUD) + `employee-availability.ts` (scheduling/handover)).
- **Business Logic**: Separated service layer with dependency injection.
- **Error Handling**: Centralized `asyncHandler` and `AppError` for consistent JSON error responses. `extractUserFriendlyDbError()` translates PostgreSQL error codes (22P02, 23505, 23503, 23502, 22003) into German user messages. Every `useMutation` MUST have `onError` with `toast({ title: "Fehler", description: error.message, variant: "destructive" })`. See `.agents/skills/error-handling-audit/SKILL.md` for full conventions.
- **Security**: Role-based access control with SQL-level data filtering, CSRF protection (including auth routes), session management, rate limiting (global API + login + password-reset), helmet security headers with Content Security Policy (CSP). Object Storage downloads authorized per-user via DB lookup (`server/middleware/object-storage-auth.ts`): admins unrestricted, employees only own docs + assigned customer docs. Centralized `sanitizeUser()` helper (`server/utils/sanitize-user.ts`) strips `passwordHash` from all API responses; `SafeUser` type exported from schema. Password validation enforces uppercase, lowercase, and digit requirements (Zod-validated on change-password route).
- **Access Model**: Three-tiered access (Superadmin → Admin → Mitarbeiter). Superadmin (`isSuperAdmin`) has full access to everything and can create/manage admins and their permissions. Admins (`isAdmin`) can only access admin areas granted via `admin_permissions` table (per-tile/area permissions configurable by superadmin). Non-admin employees see only their own appointments (filtered by `assignedEmployeeId`/`performedByEmployeeId`). Admins see all appointments. Route-level permission enforcement via `requireAdminPermission` middleware in `server/routes/admin.ts` (path-based mapping). Permission keys: users, time_entries, birthday_cards, statistics, prospects, customers, insurance_providers, documents, services, billing, hours_overview, settings, audit_log.

### Data Storage
- **Database**: PostgreSQL via Neon serverless with Drizzle ORM.
- **Schema**: Tables include `customers`, `appointments`, `insurance_providers`, `employee_time_entries`, with historization (`valid_from`/`valid_to`) and `withTimezone: true` for all timestamps.
- **Soft-Delete**: `deletedAt` for GoBD compliance on `appointments`, `customers`, `tasks`, `time_entries`, `documents`, `employee_document_proofs`, `prospects`, `monthly_service_records`, `budget_allocations`. Partial indexes on appointments for `deleted_at IS NULL` queries. All queries filter by `isNull(deletedAt)`. Service record deletion and budget allocation deletion use soft-delete (SET deletedAt) instead of hard DELETE for GoBD Aufbewahrungspflicht.
- **Data Layer**: `IStorage` interface abstraction in `server/storage.ts` (309 lines), delegating to domain modules: `server/storage/customers-storage.ts`, `server/storage/appointments-storage.ts`, `server/storage/service-records-storage.ts`, `server/storage/billing-storage.ts`, `server/storage/appointment-helpers.ts`. `budget-ledger.ts` is NOT split (~1840 lines, tightly coupled).
- **Employee Assignments**: Each customer supports 3 employee roles — 1 primary (Hauptzuständig), 1st backup (1. Vertretung), 2nd backup (2. Vertretung). All three are tracked in `customer_assignment_history` (roles: "primary", "backup", "backup2"). Access control, caching, birthday notifications, and appointment logic treat all 3 roles equally.
- **Caching**: In-memory cache for assigned customer IDs, sessions, birthdays, and company settings (60s TTL via `getCachedCompanySettings()` in `server/services/cache.ts` — auto-invalidated on update).
- **Performance**: Combined API endpoints, scoped auth middleware, single JOIN for session validation, batch cleanup. Configurable `staleTime` for frontend data stability. DB indexes: composite on `customer_service_prices(customer_id, valid_from, valid_to)`, filtered on `appointments(date) WHERE deleted_at IS NULL`, functional on `audit_log(metadata->>'customerId')`. Billing batch endpoint uses 4 batch queries instead of 3N+1 per-customer queries. WhatsApp rules use batch `INSERT ... ON CONFLICT UPDATE`. Profitability statistics use LATERAL JOINs instead of correlated subqueries.

### Business Rules & Patterns
- **Shared Domain Logic**: Single source of truth.
- **Appointment Workflow**: Status-driven field editing, overlap checking. Completed appointments can be reopened to `documenting` via `POST /api/appointments/:id/reopen` for documentation correction — reverses budget transactions (atomically in a DB transaction), clears signature, preserves all other documentation data. Blocked by Leistungsnachweis lock or month-closing (admin bypass). Frontend: "Dokumentation korrigieren" button on detail page for completed+unlocked appointments, with confirmation dialog. **Erstberatung Edit**: `PATCH /api/appointments/:id/erstberatung` updates both customer data (name, phone, email, address, pflegegrad) and appointment data (date, time, duration, employee) atomically. Edit screen mirrors the create screen 1:1. Customer `name`/`address` auto-computed by `updateCustomer()`.
- **Customer Management**: Multi-step creation with localStorage draft auto-save (key: `careconnect_customer_draft`, 24h expiry, debounced 500ms). Detailed views with 6 tabs (Übersicht, Vertrag, Dokumente, Kontakte, Budgets, Versicherung). German-specific validation (`Pflegegrad`) with historization. Deactivation tracking with predefined reasons. Inline editing of customer fields (address, phone, email, Pflegegrad, pet info, medical history, agreed services) directly within tabs, with all changes logged via `auditService`. DSGVO Art. 17 customer anonymization (`POST /admin/customers/:id/anonymize`). Erstberatungs-Kunden: Decision flow after appointment completion (convert to active customer or decline with "Kein Interesse"), decline endpoint (`POST /admin/customers/:id/decline-erstberatung`) atomically deactivates customer and updates linked prospect. **Erstberatung-Sichtbarkeit**: Kunden mit `status='erstberatung'` werden überall ausgeblendet (Kundenlisten, Mitarbeiter-Ansichten, Geburtstags-Übersicht, globale Suche) — AUSSER im Admin-Erstberatung-Tab. Backend-Filter in `getCustomers()`, `getCustomersForEmployee()`, `getCustomersPaginated()`, `searchCustomers()`. Nicht-Admin-Zugriff auf Erstberatungskunden via `/customers/:id` wird mit 403 blockiert.
- **Lead-to-Client Flow**: Prospect-based Erstberatung flow — `appointments.prospect_id` FK allows linking appointments to prospects without creating a customer. `prospect_offers` table stores wizard intermediate state. Prospect status pipeline: `qualifiziert`, `disqualifiziert`, `erstberatung_durchgeführt`, `angebot_gemacht`. Endpoints: `POST /api/appointments/prospect-erstberatung` (Erstberatung without customer), `POST /api/admin/prospects/:id/convert` (atomic prospect→customer conversion), `PATCH /api/admin/prospects/:id/qualify`, `POST /api/admin/prospects/:id/offers`, `GET /api/admin/prospects/:id/offer`, `GET /api/admin/prospects/:id/appointment-data` (returns `{ prospect, appointments }`), `GET /api/customers/:id/timeline`. `customers.convertedFromProspectId` FK links back to source prospect. Startup migration: `server/startup/migrate-erstberatung-customers.ts` (adds CHECK constraint, migrates erstberatung customers back to prospects). Frontend fully implemented with prospect conversion wizard, timeline, and workaround cleanup.
- **Document Templates**: HTML-based templates with placeholders (`{{input:Feldname}}`, company placeholders) supporting fragments or full HTML documents, billing-type associations, and versioning. Supports `documentTypeId`, `context`, `targetType`, and dual signatures. Digital document flow: select template → fill inputs → preview → sign → generate PDF.
- **PDF Generation**: Server-side HTML→PDF using Puppeteer/Chromium for GoBD-compliant documents. Digital signature capture, integrity hashing, and visible audit stamps including IP, location, and hash.
- **Insurance Providers**: Admin management, historized assignment, IK-Nummer validation.
- **Budgeting & Pricing**: Three-pot budget ledger system based on German care law, cascading allocation, FIFO for §45b, carryover budgets. Budget rules: §45b monthly allocation accumulates over year + carryover (Jan-June, any month); §39/42a annual budget resets Jan 1 (no carryover); §45a monthly only when enabled. **Default monthly amount**: 131€ (`BUDGET_45B_MAX_MONTHLY_CENTS = 13100` in `shared/domain/budgets.ts`). **Auto-Fallback**: If a budget type is enabled but no `budgetStartDate` is set (no manual Startwert), the system automatically starts allocations from January 1st of the current year — this matches the legal reality that the Entlastungsbetrag accumulates from the start of the year. Startwert (initial_balance) overrides: if set for a specific month, monthly_auto allocations start from the month AFTER the Startwert. Budget mutations (manual adjustments, reversals, type settings, preferences, initial setup) are fully audit-logged for GoBD compliance. Concurrent budget consumption is serialized per customer via `pg_advisory_xact_lock` to prevent race conditions. Budget error messages include per-pot breakdown (§45b/§39-42a/§45a availability). **Budget amount priority cascade**: `customer_budget_type_settings` (monthlyLimitCents/yearlyLimitCents) → legacy `customer_budgets` → hardcoded defaults (131€/Monat). All three budget types (§45b, §45a, §39/42a) follow this cascade. **Source naming**: Allocation source `monthly_auto` (not `monthly`); startup migration in `server/startup/migrate-budget-sources.ts` renames legacy `monthly` → `monthly_auto` at boot (idempotent, wrapped in try/catch). **Budget Reservierungen**: `getPlannedCostCents(customerId)` berechnet geschätzte Kosten aller offenen Termine (status: scheduled/in_progress/documenting, type: Kundentermin) basierend auf geplanten Services und kundenspezifischen Preisen. `BudgetSummary` enthält `plannedCents` und `availableAfterPlannedCents` (= availableCents - plannedCents). Frontend zeigt "Geplant"-Karte (orange) und warnt bei negativem Restbudget. Terminanlage zeigt Budget-Warnung als Toast wenn `availableAfterPlannedCents < 0`. Reservierungen sind keine echten Transaktionen — reine Hochrechnung für die Anzeige. **Budget-Architektur (vereinfacht)**: `ensureAllocationsGeneric()` ist die zentrale private Methode für alle drei Allokationstypen (monthly/yearly). Die drei öffentlichen Methoden sind dünne Wrapper. `resolveStartDate()` ermittelt das Startdatum je Budgettyp. `syncBudgetAllocations(customerId)` ist die zentrale Sync-Methode (Command-Query-Separation): wird aufgerufen in Routes (`/overview`, `/summary`), `createCascadeConsumption`, und Appointment-Erstellung. `getBudgetSummary*`-Methoden sind reine Lesemethoden ohne Seiteneffekte. Frontend nutzt nur `/overview` (enthält alle drei Töpfe inkl. carryover); `/summary` bleibt als Alias. Datums-Hilfsfunktionen in `shared/utils/datetime.ts` zentralisiert.
- **Budget Type Temporal Validity**: `customer_budget_type_settings` has `valid_from`/`valid_to` (date, nullable). NULL = no restriction. Cascade logic in `createCascadeConsumption` skips pots where appointment date falls outside [valid_from, valid_to]. Server validates validFrom <= validTo. Frontend shows "Gültig ab" / "Gültig bis" date pickers per pot, with expiry warning.
- **Single-Transaction Rebook**: `POST /budget/:customerId/rebook-transaction` takes `transactionId` + `targetBudgetType`, creates reversal (storno) on original pot and FIFO-consumes on target. Guards: prevents double-reversal, requires full amount coverage on target (rolls back if insufficient), includes allocationId on reversal. Frontend: per-row "Umbuchen" button (ArrowRightLeft icon) on consumption rows in TransactionList, target pot selector filtered to only enabled/valid pots. Audit event: `budget_rebook_single`.
- **Customer-Specific Pricing (Temporal)**: `customer_service_prices` table with `valid_from`/`valid_to` (inclusive dates, both boundaries included). Supports future-dated prices (e.g., price increase from Jan 1). Insert resolves overlaps: deletes exact-date duplicates, closes older records at day before new start. All consumers query with appointment date: `valid_from::date <= date AND (valid_to IS NULL OR valid_to::date >= date)`. Consumers: `billing.ts` (loads all prices, filters per appointment date in JS), `statistics.ts` (correlated subquery with ORDER BY/LIMIT 1), `budget-ledger.ts` (SQL date cast), `budget.ts` (SQL date cast). Fallback: `services.default_price_cents`. Legacy system `customer_pricing_history` (3 fixed columns) is deprecated — 0 records, kept for backward compat only.
- **Service Catalog**: Central management of services with code, name, unit type, price, VAT, `isBillable` flag, `employeeRateCents`.
- **Employee Time Tracking**: Comprehensive tracking for client/non-client work, vacation. Past entries locked for non-admins. German labor law compliance (e.g., missing break documentation with "Nachtragen" CTA). TimeEntryDialog uses Drawer on mobile, Dialog on desktop. CalendarGrid uses memoized DayCell for render performance. All non-full-day entry types (Büroarbeit, Vertrieb, Schulung, Besprechung, Sonstiges) support optional kilometer tracking for business travel. Km are shown in day detail panel and included in monthly km summary as "Sonstige Fahrten".
- **Dashboard Day View**: The dashboard (`/`) is the central daily view showing appointments AND time entries together. Hierarchy: full-day entries (Urlaub/Krankheit) as banners at top, availability slots as info strip, then a chronological timeline mixing appointments and timed entries (Büroarbeit, Pause, Vertrieb, etc.). Supports creating, editing, and deleting time entries directly from the dashboard. Respects month-closing locks and auto-generated entry protection.
- **Auto-Breaks (Daily Recalculation)**: `checkAndRecalcDailyAutoBreak(userId, date)` in `server/services/auto-breaks.ts` recalculates automatic break requirements per §4 ArbZG whenever work data changes. Triggered fire-and-forget after: time entry create/update/delete (`server/routes/time-entries.ts`), appointment update/end/reopen/delete (`server/routes/appointments.ts`), and appointment documentation save (`server/routes/appointment-documentation.ts`). Creates/updates/removes auto-generated pause entries as needed. Considers only completed/documenting appointments and non-pause time entries for work minutes. Also runs as batch during month-closing via `generateAutoBreaksForMonth`.
- **Month-Closing Workflow**: Employee-initiated month closing locks CRUD operations.
- **Task System**: Centralized page for system notices and personal tasks. Automatic §39/42a budget renewal reminder tasks created in November (2 months before expiry), assigned to admin with high priority and Dec 1 due date.
- **In-App Notifications**: `notifications` table with events: `customer_assigned`, `appointment_created`, `task_assigned`. Fire-and-forget creation via `notificationService`. Frontend: `NotificationList` component on Aufgaben page, `useUnreadCount` hook with 60s polling, badge in navigation, one-time startup toast per session. Routes: `GET /api/notifications`, `GET /api/notifications/unread-count`, `PATCH /api/notifications/:id/read`, `POST /api/notifications/mark-all-read`. Notifications only for actions done by others (no self-notifications).
- **Travel Route Calculation (OSRM)**: Automatic route distance/duration calculation for appointment travel documentation. Uses OpenStreetMap Nominatim for geocoding customer/company/employee addresses (stored as `latitude`/`longitude` in DB) and OSRM Demo API for route calculation. Geocoding happens on customer create/update (including Erstberatung create/update), company settings update, and employee profile update (background, non-blocking). Batch geocoding of missing coordinates runs at startup for customers, employees, and company. Travel suggestion endpoint (`GET /api/appointments/:id/travel-suggestion`) enriched with `suggestedKilometers`/`suggestedMinutes`. On-demand recalculation endpoint (`GET /api/appointments/:id/route-calculation?originType=home|appointment&fromAppointmentId=X`). Frontend: values auto-fill km/minutes fields; blue info banner shows calculated route; recalculates when user changes origin type.
- **Address Autocomplete**: Nominatim-based address autocomplete on all address input fields. Backend proxy `GET /api/address-search?q=<text>` queries Nominatim with `addressdetails=1`, `countrycodes=de`, rate-limited (1 req/sec). Reusable `AddressAutocomplete` component (`client/src/components/address-autocomplete.tsx`) with 500ms debounce, keyboard navigation, mobile-friendly dropdown. Integrated in: `AddressFields` (customers), Erstberatung form, employee form (`user-form.tsx`), company settings. On select: auto-fills strasse, hausnummer/nr, plz, stadt.
- **Service Records (Leistungsnachweise)**: Two types — `monthly` (traditional, all appointments in a month) and `single` (per-appointment). Both go through the same signature flow (employee → customer). Schema: `monthly_service_records.record_type` ("monthly"|"single"). API: `POST /api/service-records` (monthly), `POST /api/service-records/single` (single, body: `{customerId, appointmentId}`), `GET /api/service-records/for-appointment/:appointmentId`. Billing bundles all signed LNs for a period into one invoice. Monthly creation auto-excludes appointments already covered by single LNs. **PDF structure**: When multiple LNs exist for one invoice, each LN gets its own page with own appointment table, confirmation text, and signature pair (`page-break-before: always` between sections). `InvoicePdfData.lineItems` carry `appointmentId` and signatures carry `appointmentIds[]`/`recordType` for grouping. Fallback: single-LN invoices use the original flat layout.
- **Historical Appointment Import** (`/admin/import-appointments`): Superadmin-only feature to import historical appointments from Excel files. Parses Excel with dynamic header detection, matches customers/employees/services by name, detects duplicates (same customer+date+start time). Import creates `completed` appointments with budget consumption via `budgetLedgerStorage.createConsumptionTransaction`. Supports employee remapping for unknown staff (dropdown). After import, creates synthetic Leistungsnachweise (status=completed) per customer+employee+month. Files: `server/services/appointment-import.ts` (parser + matching + import logic), `server/routes/admin/import-appointments.ts` (API endpoints), `client/src/pages/admin/import-appointments.tsx` (UI). All imported records marked with `notes="Import aus Altdaten"`.
- **Appointment Copy**: "Termin kopieren" button on appointment detail page. Navigates to `/new-appointment?copyFrom={id}`. Pre-fills customer, services, notes, and assigned employee. Date/time defaults to today/09:00. Blue info banner "Kopie von Termin bei [Kundenname]".
- **AppointmentCard (Dashboard)**: Compact card showing time, customer name, address, service type label, color-coded left border, and status icon. Quick-access buttons for phone call and Google Maps navigation only. All management actions (edit, delete, copy, document, reopen) are on the appointment detail page — no dropdown menu or swipe actions on the card itself. Click navigates to `/appointment/:id`.
- **Customer Kilometers**: Separate tracking for travel with/for the customer.
- **Birthdays**: Integrated tab showing upcoming birthdays.
- **Employee Handover (Kunden & Termine übergeben)**: Admin feature on the users page to bulk-reassign all customer assignments (primary/backup/backup2) and future appointments from one employee to another. Button on each user card opens a dialog where admin selects the target employee. Live preview shows affected customers by role and future appointments before confirmation. Handles edge cases: if target is already assigned to a customer in a different role, that role is cleared to avoid duplicate assignments. Transaction-based with full `customer_assignment_history` tracking. Audit-logged as `employee_handover`. Routes: `GET /admin/employees/:id/handover-preview`, `POST /admin/employees/:id/handover`. Frontend: `HandoverDialog` in `client/src/pages/admin/users.tsx`.
- **Employee Self-Service Profile**: Employees manage contact data, emergency contact, pet acceptance, password, and document uploads.
- **Document Type System**: `document_types` table defines document types with `context`, `targetType`, `inputMethod` (upload/signature/both), `isMandatory`, `renewalDays`, and optional linkage to `document_template`. Trigger-based requirements via `document_type_triggers` table determine when a document type is required.
- **Trigger-Based Document Requirements**: Three trigger types: `field_match` (billingType, pflegegrad, boolean fields), `role` (employee roles), `always`. Registry in `shared/domain/document-triggers.ts`. Engine in `server/services/document-trigger-engine.ts` evaluates triggers for customers (by billingType etc.) and employees (by roles). Endpoints: `GET /api/customers/document-requirements/:billingType`, `GET /api/admin/document-requirements/employee/:employeeId`. Admin trigger CRUD: `GET/PUT /admin/document-types/:id/triggers`.
- **Document Compliance Checker**: `server/services/document-compliance-checker.ts` syncs employee document requirements based on triggers, creates pending proofs for missing documents, and checks for expiring documents based on `renewalDays`.
- **Employee Document Proofs**: `employee_document_proofs` table tracks document submission status (pending/uploaded/approved/rejected) for employee document requirements. Proof management via document storage (admin routes: `/admin/proofs/pending-review`, `/admin/proofs/:id/review`; profile routes: `/profile/proofs`, `/profile/proofs/:id/upload`).
- **Customer Documents**: Document-type-centric UI for existing documents. "Hinzufügen" allows digital creation via template or upload. Customer creation wizard (`signatures-step.tsx`) uses trigger-based requirements endpoint to determine which documents are needed per billing type, rendering signature/upload cards based on `inputMethod`. Wizard signatures step includes document preview.
- **Employee Document Upload**: Employees can upload and view customer documents (PDF, images, Word) from the customer detail page.

### Build & Deployment
- **Build Script**: `script/build.ts` — esbuild for server (CJS bundle), Vite for client. Uses an allowlist to bundle specific packages (express, zod, etc.) for faster cold starts; unlisted dependencies are external. IMPORTANT: `drizzle-orm`, `drizzle-zod`, `@neondatabase/serverless`, and `ws` must NOT be bundled — bundling drizzle-orm breaks SQL template fragment composition with nested parameters.

Both build.ts and check-build.mjs must use identical hashing logic. Critical: mtime-based hashing breaks in deployment environments where file timestamps differ.
- **Deployment Config**: `.replit` `[deployment]` section — `build` runs `npm run build`, `run` executes check-build then starts `dist/index.cjs`.
