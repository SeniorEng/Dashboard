# CareConnect - Elderly Care Service Management System

## Overview
CareConnect is a full-stack, mobile-first web application designed to streamline operations for caregivers managing elderly care services. It provides functionalities for scheduling, tracking, and documenting appointments, including digital signatures and real-time data management. The system aims to enhance efficiency for care professionals, offer comprehensive customer management, unify service models, ensure robust data historization for compliance, and adhere to German labor laws. The business vision includes expanding customer management capabilities, unifying service models, and ensuring data integrity for auditing.

## User Preferences
- Preferred communication style: Simple, everyday language
- **Keine Avatare/Profilbilder**: Für Kunden und Mitarbeiter werden keine Fotos oder Avatar-Platzhalter verwendet. Stattdessen werden Namen direkt mit Badges (z.B. Pflegegrad) dargestellt. Dies spart Platz und hält die Oberfläche aufgeräumt.
- **Keine Blur-Effekte**: Kein `backdrop-blur`, kein `bg-black/80` oder ähnlich starke Overlay-Verdunkelung. Dialog-/Sheet-/Drawer-Overlays verwenden maximal `bg-black/50` ohne Blur-Filter. Die UI soll klar und technisch scharf bleiben.
- **Keine CSS-Transforms in Overlay-Komponenten**: Dialog, AlertDialog, Sheet und Drawer dürfen KEINE `translate`, `scale`, `zoom` oder `slide` CSS-Transforms verwenden. Diese verursachen Sub-Pixel-Rendering und unscharfen Text. Stattdessen: Flexbox-Zentrierung (`fixed inset-0 flex items-center justify-center`) und reine Fade-Animationen (`fade-in-0`/`fade-out-0`, nur opacity). Drawer: `shouldScaleBackground = false`. Ausnahme: Sheet-Slide-Animationen (`slide-in-from-*`/`slide-out-to-*`) sind erlaubt, da Sheets am Bildschirmrand positioniert sind und keine Sub-Pixel-Probleme verursachen.
- **Standard-Unterschrift-Komponente**: Für ALLE Unterschriften im System MUSS die zentrale `SignaturePad`-Komponente (`@/components/ui/signature-pad.tsx`) verwendet werden. KEINE eigenen Signature-Dialoge, Canvas-Implementierungen oder alternative Unterschriftenlösungen bauen. `SignaturePad` bietet eine konsistente Fullscreen-Unterschriftserfahrung mit „Tippen zum Unterschreiben"-Platzhalter, X-Markierung und einheitlichem Styling. Wird verwendet in: Kundenanlage (signatures-step), Leistungsnachweis-Unterschrift, digitaler Dokumentenfluss.

## System Architecture

### Frontend
- **Frameworks**: React 19 with TypeScript, Vite, Wouter for routing.
- **UI/UX**: Mobile-first responsive design using `shadcn/ui` components on Radix UI primitives, styled with Tailwind CSS v4 and a "Care & Clarity" theme. Centralized `@/design-system` for consistent styling. Touch-optimized UI components. Layouts use a warm beige gradient background. Reusable `Skeleton` and `PageSkeleton` components (`@/components/ui/skeleton.tsx`) for content-shaped loading states (variants: `default`, `detail`, `list`). All `<img>` tags MUST include `width` and `height` attributes for CLS prevention.
- **State Management**: TanStack Query for data fetching. `staleTime: 30s`, `refetchOnWindowFocus: true`. Centralized query invalidation system (`@/lib/query-invalidation.ts`): `invalidateRelated(queryClient, ...domains)` maps domain dependencies (e.g., appointment mutations auto-invalidate time-entries, service-records, budget, customers, notifications, auth). All mutations MUST use `invalidateRelated()` instead of manual `queryClient.invalidateQueries()` for cross-domain consistency. Navigation prefetching via `queryClient.prefetchQuery()` on hover/focus (AppointmentCard, CustomerCard) for smoother perceived navigation.
- **Error Boundaries**: Two-tier system — global `ErrorBoundary` wraps the entire app (`App.tsx`), page-level `PageErrorBoundary` wraps every `ProtectedRoute` and `AdminRoute`. `PageErrorBoundary` shows a full-page German error message with "Seite neu laden" button. Both auto-reload on chunk load errors (code-split failures). Components in `client/src/components/error-boundary.tsx`.
- **Date/Time Handling**: German local time, no UTC conversion.
- **API Calls**: Central API client (`@/lib/api/client.ts`) for all HTTP requests, including CSRF protection, with `unwrapResult()` for data extraction. Direct `fetch()` calls are forbidden.
- **Feedback**: German toast notifications on success and error for all `useMutation` hooks.
- **Phone Number Handling** (zentral in `@shared/utils/phone`):
  - **Speicherung**: Immer E.164 (`+4917632238795`) via `normalizePhone()`.
  - **Anzeige**: Immer `formatPhoneForDisplay()` verwenden — zeigt deutsches Nationalformat (z.B. `0176 32238795`) oder internationales Format für Nicht-DE-Nummern. Niemals rohe E.164-Werte im UI anzeigen.
  - **Eingabefelder**: `formatPhoneAsYouType()` für Live-Formatierung während der Eingabe.
  - **Validierung**: `validateDachPhone()` vor dem Speichern aufrufen.
  - **tel:-Links**: `href="tel:${rawE164}"` mit dem rohen E.164-Wert, aber Anzeigetext immer via `formatPhoneForDisplay()`.
- **Adressformatierung** (zentral in `@shared/utils/format`):
  - **Anzeige**: Immer `formatAddress()` verwenden. Akzeptiert Objekte mit `strasse`, `nr`/`hausnummer`, `plz`, `stadt` und fällt auf das `address`-Feld zurück. Ausgabe: `"Strasse Nr, PLZ Stadt"`.
  - Niemals Adresskomponenten manuell im JSX zusammensetzen.
  - **Eingabe**: Immer `AddressAutocomplete` aus `@/components/address-autocomplete` für das Straßenfeld verwenden. Die Komponente nutzt Nominatim und befüllt automatisch Straße, Hausnummer, PLZ und Stadt.
  - **PLZ-Felder**: Immer `maxLength={5}`, `inputMode="numeric"` und `replace(/\D/g, "").slice(0, 5)` im onChange verwenden.
  - **Schema-Hinweis**: Kunden/Interessenten nutzen `nr` als Hausnummer-Feld, Mitarbeiter/Firma/Pflegekassen nutzen `hausnummer`. `formatAddress()` akzeptiert beides. Bei `onAddressSelect` das richtige Feld-Mapping beachten (`addr.hausnummer → nr` bei Kunden).
  - **Reusable Komponente**: `AddressFields` in `client/src/pages/admin/components/address-fields.tsx` — kapselt Straße+Nr+PLZ+Stadt mit Autocomplete für Kunden/Interessenten (nutzt `nr`-Feld).
- **Km-Formatierung** (zentral in `@shared/utils/format`):
  - **Anzeige**: Immer `formatKm()` verwenden — gibt `"1,5"` zurück (deutsches Komma-Format, 1 Dezimalstelle). Re-Export in `@/lib/utils` für Frontend-Kompatibilität.
- **Logging** (zentral in `server/index.ts`):
  - **Alle Server-Logs**: Immer die zentrale `log(message, source)` Funktion verwenden. `console.log` ist nur innerhalb der `log`-Funktion selbst erlaubt. `console.error` für echte Fehler ist weiterhin OK.
  - **Format**: `"HH:MM:SS AM/PM [source] message"` — z.B. `5:12:54 PM [startup] Pflegekassen-Import: ...`
  - **Import**: `import { log } from "../lib/log"` (oder relativer Pfad je nach Tiefe). Re-Export in `server/index.ts` für Abwärtskompatibilität.

### Backend
- **Framework**: Express.js with TypeScript.
- **API Design**: RESTful endpoints, Zod validation, structured error responses, modular routing. Large route files are split into sub-routers (e.g., `server/routes/admin/customers/` with `assignments.ts`, `budgets.ts`, `contracts.ts`, `details.ts`, `workflows.ts`; `server/routes/statistics/` with `revenue.ts`, `operations.ts`, `customers.ts`; `server/routes/customers/` with `contacts.ts`, `service-prices.ts`, `documents.ts`; `server/routes/admin/employees.ts` → `employee-users.ts` (user CRUD) + `employee-availability.ts` (scheduling/handover)).
- **Business Logic**: Separated service layer with dependency injection.
- **Error Handling**: Centralized `asyncHandler` and `AppError` for consistent JSON error responses. `extractUserFriendlyDbError()` translates PostgreSQL error codes (22P02, 23505, 23503, 23502, 22003) into German user messages. Every `useMutation` MUST have `onError` with `toast({ title: "Fehler", description: error.message, variant: "destructive" })`. See `.agents/skills/error-handling-audit/SKILL.md` for full conventions.
- **Security**: Role-based access control with SQL-level data filtering, CSRF protection (including auth routes), session management, rate limiting (global API + login + password-reset), helmet security headers with Content Security Policy (CSP). Object Storage downloads authorized per-user via DB lookup (`server/middleware/object-storage-auth.ts`): admins unrestricted, employees only own docs + assigned customer docs. Centralized `sanitizeUser()` helper (`server/utils/sanitize-user.ts`) strips `passwordHash` from all API responses; `SafeUser` type exported from schema. Password validation enforces uppercase, lowercase, and digit requirements (Zod-validated on change-password route).
- **Access Model**: Three-tiered access (Superadmin → Admin → Mitarbeiter). Superadmin (`isSuperAdmin`) has full access to everything and can create/manage admins and their permissions. Admins (`isAdmin`) can only access admin areas granted via `admin_permissions` table (per-tile/area permissions configurable by superadmin). Non-admin employees see only their own appointments (filtered by `assignedEmployeeId`/`performedByEmployeeId`). Admins see all appointments. Route-level permission enforcement via `requireAdminPermission` middleware in `server/routes/admin.ts` (path-based mapping). Permission keys: users, time_entries, birthday_cards, statistics, prospects, customers, insurance_providers, documents, services, billing, hours_overview, settings, audit_log.

### Data Storage
- **Database**: PostgreSQL via Neon serverless with Drizzle ORM. **Unique constraint naming**: When DB constraints use PostgreSQL default naming (`_key` suffix) instead of Drizzle's `_unique` suffix, the schema MUST use explicit `unique("constraint_name").on(col)` in the table builder (not inline `.unique()`) to match the existing DB constraint name. This prevents `drizzle-kit push` from trying to create duplicate constraints.
- **Schema**: Tables include `customers`, `appointments`, `insurance_providers`, `employee_time_entries`, with historization (`valid_from`/`valid_to`) and `withTimezone: true` for all timestamps.
- **Soft-Delete**: `deletedAt` for GoBD compliance on `appointments`, `customers`, `tasks`, `time_entries`, `documents`, `employee_document_proofs`, `prospects`, `monthly_service_records`, `budget_allocations`, `customer_service_prices`. Partial indexes on appointments for `deleted_at IS NULL` queries. All queries filter by `isNull(deletedAt)`. Service record deletion, budget allocation deletion, and service price deletion use soft-delete (SET deletedAt) instead of hard DELETE for GoBD Aufbewahrungspflicht.
- **Secret Encryption**: API secrets in `company_settings` (smtpPass, letterxpressApiKey, qontoSecretKey, whatsappAccessToken, twilioAuthToken) are encrypted at-rest using AES-256-GCM via `server/lib/crypto.ts`. Requires `ENCRYPTION_KEY` env var (64-char hex, 32 bytes). Encryption/decryption is transparent in `storage.ts` (encrypt on write, decrypt on read). Existing plaintext values are auto-migrated at startup via `server/startup/encrypt-company-secrets.ts`. Without `ENCRYPTION_KEY`, encryption is gracefully skipped.
- **Zod Validation**: Global German Zod error map installed via `shared/utils/zod-german.ts` (called at server startup). Provides German default messages for all Zod validation errors (type mismatches, min/max, required fields, email format). Individual schemas can still override with custom messages.
- **Budget Defaults**: When no customer-specific type-settings exist, the default cascade priority is: §45b (Entlastungsbetrag, enabled) → §45a (Umwandlung, disabled) → §39/42a (Ersatzpflege, disabled). This is consistent across engine, API, and cost-estimate.
- **Date/Time Column Types**: `customers.inaktiv_ab` uses PostgreSQL `date` (Drizzle: `date('col', { mode: 'string' })`). `invoice_line_items.appointment_date` uses `date`, `start_time`/`end_time` use `time`. All return strings to the API (YYYY-MM-DD / HH:mm:ss). Migration: `0011_funny_cable.sql`.
- **Customer Query**: `getCustomers()` supports optional `{ status, search }` parameters for DB-level filtering. Admin route passes `status` and `search` query params through. Employee route still uses `getCustomersForEmployee()`.
- **Data Layer**: `IStorage` interface abstraction in `server/storage.ts` (309 lines), delegating to domain modules: `server/storage/customers-storage.ts`, `server/storage/appointments-storage.ts`, `server/storage/service-records-storage.ts`, `server/storage/billing-storage.ts`, `server/storage/appointment-helpers.ts`. Budget ledger split into 8 sub-modules under `server/storage/budget/` (facade in `budget-ledger.ts`).

### Business Rules & Patterns
- **Shared Domain Logic**: Single source of truth. **Shared API Response Types** (`shared/api/`): Domain-specific response type definitions shared between frontend and backend. Files: `pagination.ts`, `employees.ts`, `customers.ts`, `insurance.ts`, `time-tracking.ts`, `index.ts`. Frontend re-exports from `@shared/api` via `client/src/lib/api/types.ts`. Canonical definitions always in `shared/api/`.
- **Fahrtdienst (Transport/Doctor Escort)**: Planning sub-type of Alltagsbegleitung. When an Alltagsbegleitung service is selected, a "Fahrtdienst" toggle appears. Captures doctor appointment time, doctor name (optional), and doctor address. Auto-calculates travel time via OSRM routing + buffer (10 min base, +10 min rush hour 7–9h/16–18h, +15% if >30 min). Sets `scheduledStart` to pickup time. Schema fields: `isFahrtdienst`, `doctorName`, `doctorAppointmentTime`, `doctorStrasse/Plz/Stadt`, `doctorLatitude/Longitude`, `estimatedTravelMinutes`, `travelBufferMinutes`. Backend: `server/services/travel-time.ts` + `GET /api/travel-time`. Frontend: `FahrtdienstPanel` component. Display: car icon on appointment cards, full detail panel with pickup time, travel time, buffer, and doctor info. Billing unchanged — always billed as normal Alltagsbegleitung.
- **Appointment Workflow**: Status-driven field editing, overlap checking. Completed appointments can be reopened to `documenting` via `POST /api/appointments/:id/reopen` for documentation correction — reverses budget transactions (atomically in a DB transaction), clears signature, preserves all other documentation data. Blocked by Leistungsnachweis lock or month-closing (admin bypass). Frontend: "Dokumentation korrigieren" button on detail page for completed+unlocked appointments, with confirmation dialog. **Erstberatung Edit**: `PATCH /api/appointments/:id/erstberatung` updates both customer data (name, phone, email, address, pflegegrad) and appointment data (date, time, duration, employee) atomically. Edit screen mirrors the create screen 1:1. Customer `name`/`address` auto-computed by `updateCustomer()`.
- **Customer Management**: Multi-step creation with localStorage draft auto-save (key: `careconnect_customer_draft`, 24h expiry, debounced 500ms). Detailed views with 6 tabs (Übersicht, Vertrag, Dokumente, Kontakte, Budgets, Versicherung). German-specific validation (`Pflegegrad`) with historization. Deactivation tracking with predefined reasons. Inline editing of customer fields (address, phone, email, Pflegegrad, pet info, medical history, agreed services) directly within tabs, with all changes logged via `auditService`. DSGVO Art. 17 customer anonymization (`POST /admin/customers/:id/anonymize`). Erstberatungs-Kunden: Decision flow after appointment completion (convert to active customer or decline with "Kein Interesse"), decline endpoint (`POST /admin/customers/:id/decline-erstberatung`) atomically deactivates customer and updates linked prospect. **Erstberatung-Sichtbarkeit**: Kunden mit `status='erstberatung'` werden überall ausgeblendet (Kundenlisten, Mitarbeiter-Ansichten, Geburtstags-Übersicht, globale Suche) — AUSSER im Admin-Erstberatung-Tab. Backend-Filter in `getCustomers()`, `getCustomersForEmployee()`, `getCustomersPaginated()`, `searchCustomers()`. Nicht-Admin-Zugriff auf Erstberatungskunden via `/customers/:id` wird mit 403 blockiert.
- **Lead-to-Client Flow**: Prospect-based Erstberatung flow — `appointments.prospect_id` FK allows linking appointments to prospects without creating a customer. `prospect_offers` table stores wizard intermediate state. Prospect status pipeline: 9 statuses (`neu`, `kontaktiert`, `wiedervorlage`, `qualifiziert`, `disqualifiziert`, `erstberatung_vereinbart`, `erstberatung_durchgeführt`, `gewonnen`, `nicht_interessiert`). Removed statuses `angebot_gemacht` and `absage` (migrated at startup). **Auto-status**: When an Erstberatung appointment is documented (completed), the linked prospect automatically transitions from `erstberatung_vereinbart` → `erstberatung_durchgeführt`. **Auto-matching**: At startup, prospects are matched against customers by `vorname` + `nachname` + `telefon` (normalized E.164); matches set prospect to `gewonnen` with `convertedCustomerId`. Also runs when creating a new customer. **Clickable pipeline stats**: Status boxes in the pipeline overview are clickable to filter the list; click again to reset. Endpoints: `POST /api/appointments/prospect-erstberatung` (Erstberatung without customer), `PATCH /api/admin/prospects/:id/qualify`, `POST /api/admin/prospects/:id/offers`, `GET /api/admin/prospects/:id/offer`, `GET /api/admin/prospects/:id/appointment-data` (returns `{ prospect, appointments }`), `GET /api/customers/:id/timeline`. `customers.convertedFromProspectId` FK links back to source prospect. Startup migrations: `server/startup/migrate-erstberatung-customers.ts`, `server/startup/migrate-prospect-statuses.ts` (migrates `angebot_gemacht` → `erstberatung_durchgeführt`, `absage` → `nicht_interessiert`), `server/startup/prospect-customer-matching.ts`.
- **Document Templates**: HTML-based templates with placeholders (`{{input:Feldname}}`, company placeholders) supporting fragments or full HTML documents, billing-type associations, and versioning. Supports `documentTypeId`, `context`, `targetType`, and dual signatures. Digital document flow: select template → fill inputs → preview → sign → generate PDF. **Info Input Method** (`inputMethod: "info"`): Document types like AGB or Preisliste that become automatic contract components — no upload or signature required. In the wizard, info docs show as auto-fulfilled (green checkmark) with "Automatisch Vertragsbestandteil" badge and optional preview. On customer creation/prospect conversion, `generateInfoDocumentPdfs()` auto-generates PDFs for all triggered info doc types with templates (background task). In customer documents section, info docs show "Zur Kenntnisnahme" badge and are excluded from the manual "add document" list.
- **PDF Generation**: Server-side HTML→PDF using Puppeteer/Chromium for GoBD-compliant documents. Digital signature capture, integrity hashing, and visible audit stamps including IP, location, and hash.
- **Insurance Providers**: Admin management, historized assignment, IK-Nummer validation.
- **Budgeting & Pricing**: Three-pot budget ledger system based on German care law, cascading allocation, FIFO for §45b, carryover budgets. Budget rules: §45b monthly allocation accumulates over year + carryover (Jan-June, any month); §39/42a annual budget resets Jan 1 (no carryover); §45a monthly only when enabled. **Default monthly amount**: 131€ (`BUDGET_45B_MAX_MONTHLY_CENTS = 13100` in `shared/domain/budgets.ts`). **Auto-Fallback**: If a budget type is enabled but no `budgetStartDate` is set (no manual Startwert), the system automatically starts allocations from January 1st of the current year — this matches the legal reality that the Entlastungsbetrag accumulates from the start of the year. Startwert (initial_balance) overrides: if set for a specific month, monthly_auto allocations start from the month AFTER the Startwert. Budget mutations (manual adjustments, reversals, type settings, preferences, initial setup) are fully audit-logged for GoBD compliance. Concurrent budget consumption is serialized per customer via `pg_advisory_xact_lock` to prevent race conditions. Budget error messages include per-pot breakdown (§45b/§39-42a/§45a availability). **Budget amount priority cascade**: `customer_budget_type_settings` (monthlyLimitCents/yearlyLimitCents) → legacy `customer_budgets` → hardcoded defaults (131€/Monat). All three budget types (§45b, §45a, §39/42a) follow this cascade. **Source naming**: Allocation source `monthly_auto` (not `monthly`); startup migration in `server/startup/migrate-budget-sources.ts` renames legacy `monthly` → `monthly_auto` at boot (idempotent, wrapped in try/catch). **Budget Reservierungen**: `getPlannedCostCents(customerId)` berechnet geschätzte Kosten aller offenen Termine (status: scheduled/in_progress/documenting, type: Kundentermin) basierend auf geplanten Services und kundenspezifischen Preisen. `BudgetSummary` enthält `plannedCents` und `availableAfterPlannedCents` (= availableCents - plannedCents). Frontend zeigt "Geplant"-Karte (orange) und warnt bei negativem Restbudget. Terminanlage zeigt Budget-Warnung als Toast wenn `availableAfterPlannedCents < 0`. Reservierungen sind keine echten Transaktionen — reine Hochrechnung für die Anzeige. **Budget-Architektur (vereinfacht)**: Allokationen werden BERECHNET statt materialisiert — `calculateAllocatedCents(customerId, budgetType, opts)` ist eine reine Berechnungsfunktion, die den zugewiesenen Betrag basierend auf Startdatum, Pflegegrad-Beträgen, validFrom/validTo und initial_balance-Einträgen ermittelt. Nur Carryover- und initial_balance-Allokationen werden tatsächlich in der DB gespeichert. `syncCarryoverAndExpiry(customerId)` ersetzt das alte `syncBudgetAllocations` und führt nur noch Übertrag-Erstellung + Verfall-Abschreibung durch. `getBudgetSummary*`-Methoden sind reine Lesemethoden ohne Seiteneffekte. Frontend nutzt nur `/overview` (enthält alle drei Töpfe inkl. carryover); `/summary` bleibt als Alias. Datums-Hilfsfunktionen in `shared/utils/datetime.ts` zentralisiert.
- **§45b Auto-Renewal-Modell (virtuell)**: Die monatliche §45b-Aufstockung (Default 131 €/Monat) wird **NICHT als DB-Zeilen gespeichert**, sondern bei jedem `calculateAllocatedCents`-Aufruf rein rechnerisch ermittelt (`allocation-storage.ts: calculateAllocated45b`). ### Build & Deployment
- **Build Script**: `script/build.ts` — esbuild for server (CJS bundle), Vite for client. Uses an allowlist to bundle specific packages (express, zod, etc.) for faster cold starts; unlisted dependencies are external. IMPORTANT: `drizzle-orm`, `drizzle-zod`, `@neondatabase/serverless`, and `ws` must NOT be bundled — bundling drizzle-orm breaks SQL template fragment composition with nested parameters.
