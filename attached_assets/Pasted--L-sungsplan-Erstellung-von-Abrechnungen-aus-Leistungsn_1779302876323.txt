## Verbleibende Schwachstellen (im aktuellen Code, post-Fixes)

### Tier A — wahrscheinliche Ursachen für "regelmäßig fehlt"

**A1 — Background-PDF schlägt silent fehl bei `POST /generate`**
`server/routes/billing.ts:1287` — `schedulePdfPersistInBackground()` läuft via `setImmediate` nach dem HTTP-Response. Fehler werden nur geloggt. Folge: User sieht "Rechnung erstellt", aber spätere `GET /:id/pdf`-Aufrufe scheitern, weil der Backfill nie durchgelaufen ist (z. B. Chromium-Race auf Startup, Object-Storage-Hänger, OOM beim ersten Page-Render). Diagnose erschwert, weil der Erstellungs-Endpunkt 200 OK liefert.

**A2 — Send-Endpoints umgehen den PDF-Cache**
`server/routes/billing.ts:2195-2230` (`POST /:id/send`) und `Z.495-512` (`POST /send-batch`) rendern PDFs **immer frisch** statt den persistierten Cache aus Task #521/544 zu nutzen. Konsequenzen:
- Bei Pflegekassen-Stapel mit >10 Rechnungen → N × 5-30s Puppeteer-Renders → **Replit 60s-Request-Timeout** → Batch bricht mittendrin ab, vorherige Rechnungen schon auf `versendet`
- Inkonsistenz zur GoBD-Strategie: gerade versendete PDF ist möglicherweise andere als die später aus Cache angezeigte (Drift möglich)
- Hoher Lastpunkt für Browser-Singleton → "main frame too early"-Race trotz Warmup

**A3 — `persistInvoicePdf()` race auf stale Invoice-Objekt**
`server/routes/billing.ts:1830` prüft `invoice.pdfPath`. Bei Neon-Read-Replika-Lag liest der Background-Job nach Insert noch den alten Stand mit `pdfPath=null`. Wenn der User parallel `/pdf` aufruft, läuft `persistInvoicePdf()` **doppelt** (einmal Background, einmal On-Demand) → race conditions im Object-Storage-Write, möglicherweise korrupte PDF / verlorene Updates.

**A4 — Neon-Replika-Lag nicht serverseitig abgesichert**
`client/src/lib/query-invalidation.ts:156-175` (`refetchWithPoll`) polled 6×400 ms = 2,4 s. Bei Spitzenlast wurde im Produktivbetrieb >2,4 s Lag beobachtet (siehe Commit-Message Task #543). Folge: nach "Alle offenen erstellen" zeigt die Liste nicht alle neuen Rechnungen, User klickt erneut, `/generate-all` ist zwar idempotent, aber UX wirkt wie "Fehlschlag".

**A5 — ZUGFeRD Silent Fallback erhalten**
`server/lib/zugferd.ts:285-310` — bei XML-Validierungs- oder Embedding-Fehler wird das Standard-PDF zurückgegeben, nur geloggt. `zugferdXml`-Spalte bleibt leer, Integrity-Verifier kann den Fehler nicht später entdecken. Pflegekassen, die ZUGFeRD/XRechnung verpflichtend fordern, lehnen ab → User-Wahrnehmung: "Versand klappt, aber Kasse zahlt nicht".

### Tier B — sekundäre Risiken

**B1 — Keine FOR-UPDATE-Locks auf Send-Endpoints**
`Z.2088` (`/send`), `Z.2535` (`/send-bulk`): Status-Prüfung ohne `FOR UPDATE` → zwei parallele Admins können beide `entwurf` sehen, beide senden → doppelter Versand, doppelter Delivery-Record. Storno-Pfad nutzt den Lock korrekt (`Z.1354` mit `getInvoiceForUpdateTx`), die Send-Pfade aber nicht.

**B2 — Budget-Split auf Replika-Lag empfindlich**
`Z.981` `getBudgetSplitForAppointments()` liest `budget_transactions` ohne Konsistenz-Garantie. Wenn der zugrundeliegende Budget-Eintrag gerade erst geschrieben wurde, kann Lag dazu führen, dass Kasse-Anteile fälschlich als Privat klassifiziert werden → Split-Rechnungen falsch.

**B3 — Insurance-Lookup mit `.limit(1)` ohne ORDER BY**
Frühere Stellen in `getInsuranceData()` (Z.737 vorher) noch zu prüfen, ob auf neuem Stand `ORDER BY validFrom DESC` ergänzt wurde. Falls nicht: nicht-deterministische Kasse bei mehreren aktiven Einträgen.

**B4 — Fehlende Service-Preise blockieren ganzen Rechnungslauf**
`buildLineItemsFromAppointments` wirft bei fehlendem Preis hart → ganze Massenerstellung kann an einer einzelnen Lücke scheitern. Pre-Flight-Validation pro Kunde wäre robuster: nur betroffene Kunden skippen.

### Tier C — Backlog / niedriges Risiko

- C1: Object-Storage-Write hat keine Retry-Logik in `persistInvoicePdf()` (Z.1851)
- C2: `renderLeistungsnachweisOnTheFly()` Fehler → Hard Fail in `/bundle` (Z.2447) — keine Fallback-Strategie
- C3: Stale `zugferdXml` wird nie erneuert; Integrity-Verifier loggt nur

## Best-Practice-Referenzen

1. **PDF-Pipeline auf Replit:** Background-Erzeugung sollte einen Status in der DB führen (`pdf_status: pending|ready|failed|retrying`), damit das Frontend visualisieren kann, ob ein PDF noch generiert wird. Aktuell ist es entweder da oder nicht (`pdfPath` null).
2. **Send vs. View:** Versand-Pipeline sollte **denselben** PDF-Cache nutzen wie View. Andernfalls riskiert man inkonsistente Dokumente und doppelte CPU-Last.
3. **Concurrency-Locks:** `SELECT … FOR UPDATE` ist in PostgreSQL die etablierte Lösung für "status-machine" Operationen. Drizzle: `.for("update")`. Im Codebase bereits an einer Stelle korrekt (`getInvoiceForUpdateTx`).
4. **Neon-Replika-Lag:** Anstelle reiner Frontend-Polls: nach Write `await readUntilVisible(...)` server-side oder konsistente Reads im selben Tx-Kontext.
5. **ZUGFeRD/XRechnung:** Pflicht in DE seit 01.01.2025 für B2B. Silent Fallback ist gefährlich. Best Practice: explizites `zugferd_status`-Feld plus UI-Warnung bei Failed-State.

### Phase 1 — Hauptverdächtige (P0)

**Schritt 1.1 — Background-PDF-Status persistieren**
- Datei: `shared/schema/billing.ts` (Migration ergänzen) + `server/routes/billing.ts:1287`
- Neue Spalte `pdf_status` enum (`pending | generating | ready | failed`) + `pdf_last_error text`
- `schedulePdfPersistInBackground`: Status-Updates schreiben (`pending → generating → ready/failed`)
- Frontend: in der Rechnungs-Liste sichtbar (Badge oder Spinner), bei `failed` Retry-Knopf
- Konsequenz: User merkt sofort, ob ein PDF erzeugt wurde, ohne den Download-Versuch

**Schritt 1.2 — Send-Endpoints auf PDF-Cache umstellen**
- Datei: `server/routes/billing.ts:2195-2230` (`/send`) und `Z.495-512` (`/send-batch`)
- Vor dem Versand: `persistInvoicePdf(id)` aufrufen → garantiert Cache vorhanden → dann `loadStoredPdfByPath()` aus Object Storage statt frisch zu rendern
- Bei Pflegekasse mit Beihilfe: gespeichertes PDF laden, dann via pdf-lib verdoppeln
- Effekt: Stapelversand 50 Rechnungen reduziert sich auf Storage-Reads + SMTP, eliminiert 60s-Timeout-Risiko und Browser-Race
- Konsistenz: Versendetes PDF ist garantiert identisch mit dem im View angezeigten

**Schritt 1.3 — FOR-UPDATE-Locks auf Send-Endpoints**
- Datei: `server/routes/billing.ts:2084-2127` (`/send`), `Z.2532-2569` (`/send-bulk`), `Z.415-490` (`/send-batch`)
- Status-Check + Update in `db.transaction` mit `getInvoiceForUpdateTx(tx, id)` wrappen (existiert bereits in `billing-storage.ts`)
- Verhindert Doppelversand bei parallelen Klicks

**Schritt 1.4 — `persistInvoicePdf()` doppelt-laufen verhindern**
- Datei: `server/routes/billing.ts:1818-1917`
- Either: pro-invoice Mutex (in-memory Map mit `Promise`), beim zweiten Aufruf wird auf den ersten gewartet
- Or: PostgreSQL advisory lock `pg_advisory_xact_lock(hashtext('persist_pdf_' || invoice_id))`
- Effekt: Background-Job + On-Demand-Aufruf laufen seriell, kein Object-Storage-Race

### Phase 2 — Härtung (P1)

**Schritt 2.1 — ZUGFeRD fail loud**
- Datei: `server/lib/zugferd.ts:285-310`
- Neue Invoice-Spalte `zugferd_status` (`ok | validation_failed | embedding_failed | skipped`) + `zugferd_error text`
- Status setzen bei jedem Pfad in `embedZugferdXml()`
- Frontend zeigt Warnsymbol an Rechnungen mit `zugferd_status != "ok"`, UI-Hinweis vor Versand

**Schritt 2.2 — Neon-Replika-Lag serverseitig abfangen für `generate-all`**
- Datei: `server/routes/billing.ts:2636-2722`
- Nach jedem `generateInvoiceCore()`-Aufruf die Invoice-ID in die Response aufnehmen
- Frontend nutzt die IDs direkt zur Anzeige (keine Liste-Refetch nötig), Server-Truth ist die Response
- Alternative: `Cache-Control: no-store` + `?include=<ids>` Query-Param für gezielte Refetches mit längerem Poll-Timeout

**Schritt 2.3 — Insurance-Lookup deterministisch**
- Datei: `server/routes/billing.ts` (alle `customerInsuranceHistory`-Queries mit `.limit(1)` ohne ORDER BY)
- Ergänzen: `.orderBy(desc(customerInsuranceHistory.validFrom), desc(customerInsuranceHistory.id))`
- Wiederverwendbare Helper-Funktion `getActiveInsurance(customerId)` im Storage-Layer

**Schritt 2.4 — Pre-Flight für `generate-all`**
- Datei: `server/routes/billing.ts:2636-2722`
- Vor der Schleife: Service-Katalog-Vollständigkeit + Versicherungs-Aktivität pro Kunde prüfen
- Kunden mit fehlenden Voraussetzungen in `skipped[]` mit klarem `reason` melden, nicht erst beim Rendering scheitern

### Phase 3 — Tests & Beobachtung (P2)

**Schritt 3.1 — Edge-Case-Tests für /generate-all + Background-PDF**
- Datei: `tests/billing/billing-flow.test.ts` erweitern
- Cases: Chromium fehlt zum Erstellungs-Zeitpunkt; Object-Storage hängt; parallele `persistInvoicePdf`; Send mit fehlendem Cache; ZUGFeRD-Validation-Failure; Customer ohne aktive Kasse
- Helper aus bestehender Suite wiederverwenden

**Schritt 3.2 — Smoke-Script für PDF-Pipeline**

- Erzeugt Test-Rechnung, ruft `persistInvoicePdf` direkt auf, validiert Storage-Write + Hash, validiert ZUGFeRD-XML

### Phase 4 — Optional / Backlog
