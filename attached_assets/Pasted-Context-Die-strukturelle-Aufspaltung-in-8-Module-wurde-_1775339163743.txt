Context
Die strukturelle Aufspaltung in 8 Module wurde bereits gemacht (Commit 1522a36). budget-ledger.ts ist jetzt eine 106-Zeilen-Facade. Aber die inhaltliche Komplexitaet steckt unveraendert in den Sub-Modulen -- insgesamt 2.251 Zeilen. Groesster Problembereich: allocation-storage.ts mit 716 Zeilen, wovon ~500 Zeilen Auto-Allokations-Materialisierung sind.

Wo liegt die unnoetige Komplexitaet?
1. Auto-Allokationen als DB-Zeilen (allocation-storage.ts, ~500 von 716 Zeilen)
Fuer JEDEN Monat wird eine budget_allocations-Zeile erzeugt (source: "monthly_auto"). Das erfordert:

ensureAllocationsGeneric (Z.162-250, 90 Zeilen) -- generische Insert-Schleife
ensureMonthlyAllocations (Z.266-383, 118 Zeilen) -- §45b mit 4-stufiger Startdatum-Fallback-Kette, Initial-Balance-Erkennung, validFrom/validTo-Eingrenzung
ensureAllocations45a (Z.385-433, 49 Zeilen) -- §45a Variante
ensureAllocations39_42a (Z.435-456, 22 Zeilen) -- §39/42a Variante
resolveStartDate (Z.252-264, 13 Zeilen) -- Startdatum-Fallback
softDeleteOutOfRangeAllocations (Z.664-697, 34 Zeilen) -- Aufraeum-Logik fuer ungueltige Allokationen
isAllocationOutOfRange (Z.629-662, 34 Zeilen) -- Validierung
syncBudgetAllocations (Z.699-716, 18 Zeilen) -- Orchestrierung aller ensure*-Methoden

Kernproblem: Diese Zeilen sind deterministisch -- der Monatsbetrag, das Startdatum, und der Budget-Typ reichen aus um den Gesamtbetrag zu BERECHNEN. Stattdessen wird er materialisiert, was Sync-Logik, Soft-Delete, N+1-Inserts und syncBudgetAllocations-Aufrufe bei jedem Request erzwingt.
2. FIFO auf Allokations-Ebene (consumption-engine.ts, ~130 Zeilen)
consumeFifo (Z.18-148) iteriert ueber alle Allokations-Zeilen und trackt Verbrauch pro Allokation. Das ist nur fuer Carryover relevant (hat Ablaufdatum). Regulaere Monatsbudgets verfallen nicht innerhalb des Jahres -- FIFO bringt dort keinen Mehrwert.
3. getBudgetSummary mit 6 parallelen Queries (summary-queries.ts, ~120 Zeilen)
getBudgetSummary (Z.106-230) laeuft 6 parallele DB-Queries:

Summe aller Allokationen (die man berechnen koennte)
Transaktionen gruppiert nach Typ
Allokationen des aktuellen Jahres
Carryover-Allokationen
Verbrauch des aktuellen Monats
Preferences

Queries 1 und 3 waeren unnoetig wenn Allokationen berechnet statt abgefragt wuerden.
4. ensureYearlyCarryover45b (allocation-storage.ts, ~95 Zeilen)
Berechnet pro Jahr: totalAllocated - netConsumed = unused -> carryover. Richtige Logik, aber abhaengig davon dass alle monatlichen Allokations-Zeilen existieren. Mit berechneten Allokationen wuerde sich das vereinfachen.

Eliminiert: ensureAllocationsGeneric, ensureMonthlyAllocations, ensureAllocations45a, ensureAllocations39_42a, resolveStartDate, softDeleteOutOfRangeAllocations, isAllocationOutOfRange (~360 Zeilen)
Schritt 2: syncBudgetAllocations -> syncCarryoverAndExpiry
Reduziert auf:
typescriptexport async function syncCarryoverAndExpiry(customerId: number, _tx?: DbClient): Promise<void> {
  await ensureYearlyCarryover45b(customerId, _tx);
  await processExpiredCarryover(customerId, _tx);
}
Alle 6+ Aufrufer (budget.ts, appointments.ts, appointment-series.ts, admin/customers.ts, startup/sync-budget-allocations.ts) werden umgestellt.
Eliminiert: Die 3x parallelen ensure*-Aufrufe + softDeleteOutOfRangeAllocations
Schritt 3: getBudgetSummary vereinfachen
Statt 6 parallele Queries:

    // Preferences (bleibt)
  ]);

}
Aehnlich fuer getBudgetSummary45a und getBudgetSummary39_42a.
Schritt 4: consumeFifo vereinfachen
Aktuell iteriert FIFO ueber alle Allokations-Zeilen. Vereinfacht:

Carryover-Allokationen zuerst (haben Ablaufdatum, FIFO bleibt noetig -- aber nur ueber wenige Zeilen)
Regulaeres Budget: Direkt als Transaktion buchen, allocationId = null (kein FIFO noetig, berechnet statt materialisiert)

Alternativ, konservativer: FIFO-Logik behalten aber nur ueber Sonder-Allokationen (carryover, initial_balance) laufen lassen, nicht ueber Dutzende monthly_auto-Zeilen.
Schritt 5: ensureYearlyCarryover45b anpassen
Statt ueber alle monatlichen Allokations-Zeilen zu iterieren:
typescript// Alt: totalAllocated = SUM(allocation rows for year)
// Neu: totalAllocated = calculateAllocatedCents(customerId, "45b", { year })
// Dann: unused = totalAllocated - netConsumed (aus Transaktionen)
Spart ~30 Zeilen und die Abhaengigkeit von materialisierten Allokationen.
Schritt 6: Legacy-monthly_auto-Zeilen

Bestehende monthly_auto Zeilen in der DB bleiben (keine destruktive Migration)
Neue Logik ignoriert sie fuer Berechnungen
Bestehende Transaktionen mit allocationId auf alte Allokationen bleiben valide
Optional spaeter: Aufraeum-Migration die alte monthly_auto Zeilen soft-deleted
