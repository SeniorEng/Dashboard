## 🔒 Task 2: Zod Validierung härten (Security & Datenqualität)
**Kontext:** Einige Zod-Schemas sind noch zu generisch und könnten ungültige Daten durchlassen.

**To-Do:**
1. **Passwort-Sicherheit (`shared/schema/users.ts`):** Passe das `insertUserSchema` und `passwordResetSchema` an. Ändere `z.string().min(8)` in eine strengere Validierung (mindestens 8 Zeichen, ein Großbuchstabe, eine Zahl).
   *Beispiel:* `.regex(/^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)[a-zA-Z\d\w\W]{8,}$/, "Passwort muss Groß-/Kleinbuchstaben und eine Zahl enthalten")`
2. **Zeit-Validierung (`shared/schema/appointments.ts`):** Im `insertKundenterminSchema` und `insertErstberatungSchema` ist `scheduledStart` aktuell nur `z.string()`. Bitte füge hier denselben Regex-Check hinzu, der bereits im `documentAppointmentSchema` für `actualStart` verwendet wird: `.regex(/^\d{2}:\d{2}(:\d{2})?$/, "Ungültiges Zeitformat (HH:MM erwartet)")`.

## 🛡️ Task 3: Passwort-Hashes absichern (Security Pitfall)
**Kontext:** In `shared/schema/users.ts` liegt `passwordHash` direkt in der `users` Tabelle. Das Risiko ist hoch, dass bei einem `SELECT * FROM users` (oder `db.select().from(users)`) der Hash ans Frontend geleitet wird, wenn Entwickler vergessen, ihn manuell auszuschließen.

**To-Do:**
1. Erstelle in `shared/schema/users.ts` ein sicheres Zod-Schema für den Frontend-Export: `export const safeUserSchema = createSelectSchema(users).omit({ passwordHash: true });`
2. Lege den Typen an: `export type SafeUser = z.infer<typeof safeUserSchema>;`
3. Durchsuche das Backend (insbesondere `server/services/auth.ts`, falls vorhanden, oder API-Routen) und stelle sicher, dass bei API-Antworten, die User-Objekte zurückgeben, der `passwordHash` immer via Code (z.B. Destructuring `const { passwordHash, ...safeUser } = user;`) entfernt wird. Nutze dafür eine Hilfsfunktion `sanitizeUser`.
