TEIL 2: Performance & Refactoring Assessment
Kontext
Basierend auf aktuellen Best Practices (React 19, TanStack Query v5, Express.js, Drizzle ORM, Vite 6) wurde der gesamte Code auf Performance-Probleme, Anti-Patterns und Refactoring-Bedarf analysiert. Die Befunde sind nach Impact geordnet.

A. FRONTEND PERFORMANCE

A1. KRITISCH: staleTime: Infinity verhindert Daten-Aktualisierung
Datei: client/src/lib/queryClient.ts, Zeile 93
Problem: Die globale QueryClient-Konfiguration setzt staleTime: Infinity. Das bedeutet: Daten werden nie als veraltet betrachtet und nie automatisch neu geladen — auch nicht beim Zurücknavigieren zu einer Seite. Wenn Mitarbeiter A einen Termin dokumentiert, sieht Mitarbeiter B die Änderung erst nach manuellem Neuladen.
Einzelne Hooks überschreiben das zwar teilweise (z.B. useCustomers mit staleTime: 30000), aber viele Queries nutzen den globalen Default und werden nie aktualisiert.
Best Practice (TanStack Query v5): Globaler Default sollte 1-5 Minuten sein, nicht Infinity.
Anweisung:
typescript// client/src/lib/queryClient.ts, Zeile 87-100
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      queryFn: getQueryFn({ on401: "throw" }),
      refetchInterval: false,
      refetchOnWindowFocus: true,     // WAR: false → Daten aktualisieren bei Tab-Wechsel
      staleTime: 2 * 60 * 1000,      // WAR: Infinity → 2 Minuten
      gcTime: 10 * 60 * 1000,        // NEU: Garbage Collection nach 10 Min
      retry: 1,                       // WAR: false → 1 Retry bei Netzwerkfehlern
    },
    mutations: {
      retry: false,
    },
  },
});
Auswirkung: Alle Seiten zeigen automatisch aktuelle Daten nach 2 Minuten. Besonders wichtig für den Dashboard-Terminkalender und die Leistungsnachweis-Übersicht.

A2. HOCH: Inline-Objekte und -Funktionen verursachen unnötige Re-Renders
Dateien: Mehrere Seiten
Problem: Inline-Objekte und -Funktionen in JSX erzeugen bei jedem Render neue Referenzen, was unnötige Re-Renders von Kind-Komponenten auslöst.
Beispiele gefunden:

customer-new.tsx Zeile 115: setLocation(/admin/customers/${customer.id}) — inline Arrow-Function in onSuccess ist hier OK (Event-Handler), aber:
customer-detail.tsx Zeile 415:

tsxonTogglePrivatePayment={(checked) => togglePrivatePayment.mutate(checked)}
Wird bei jedem Render der Elternkomponente neu erstellt → BudgetsTabContent rendert unnötig neu.

dashboard.tsx Zeile 135:

tsxonSelect={setSelectedDate}
Das ist korrekt (stabile Referenz). Aber weekDays.map(...) erzeugt neue Day-Objekte.

customer-edit.tsx Zeile 541-542:

tsxonCheckedChange={(checked) => setFormData(prev => ({ ...prev, haustierVorhanden: checked }))}
Inline-Callbacks für jeden Switch — bei 5+ Switches = 5+ neue Funktionen pro Render.
Anweisung:

Extrahiere häufig genutzte Inline-Handler mit useCallback:

typescript// customer-detail.tsx
const handleTogglePrivatePayment = useCallback(
  (checked: boolean) => togglePrivatePayment.mutate(checked),
  [togglePrivatePayment]
);

Für Formulare mit vielen Switches: Erstelle einen generischen Handler:

typescriptconst handleSwitchChange = useCallback(
  (field: string) => (checked: boolean) => {
    setFormData(prev => ({ ...prev, [field]: checked }));
  },
  []
);
// Nutzung: onCheckedChange={handleSwitchChange("haustierVorhanden")}
Hinweis: Der React 19 Compiler kann viele dieser Optimierungen automatisch übernehmen (siehe A6).

A3. HOCH: Dashboard-Komponente ohne Memoization für teure Berechnungen
Datei: client/src/pages/dashboard.tsx
Problem: weekDays (Zeile 78) ist korrekt memoisiert, aber die Appointment-Zählung pro Tag (weekAppointmentCounts) macht einen separaten API-Call pro Wochenwechsel. Das ist an sich OK, aber die Kombination mit staleTime: Infinity bedeutet: Einmal geladene Wochen werden nie aktualisiert, auch nicht beim Zurücknavigieren.
Anweisung:

Entferne die expliziten staleTime-Overrides in den Dashboard-Hooks und verlasse dich auf den neuen globalen Default (2 Min)
Die weekAppointmentCounts Query sollte ein kürzeres staleTime haben (30 Sekunden), da Termine häufig geändert werden


A4. HOCH: Routing erzeugt neue Komponenten-Instanzen bei jedem Render
Datei: client/src/App.tsx, Zeilen 114-145
Problem: Mehrere Routen nutzen ein Anti-Pattern, das bei jedem Render eine neue Komponente erzeugt:
tsx// PROBLEM: Zeile 115
<Route path="/customer/:id">
  {() => <ProtectedRoute component={() => <CustomerDetail />} />}
</Route>

// PROBLEM: Zeile 133
<Route path="/service-records/:id">
  {() => <ProtectedRoute component={() => <ServiceRecordDetailPage />} />}
</Route>
Das component={() => <CustomerDetail />} erzeugt bei jedem Render eine neue anonyme Komponente. React sieht eine neue Komponente → unmountet die alte → mountet die neue → verliert den gesamten State und startet alle Queries neu.
Anweisung:
tsx// VORHER (schlecht):
<Route path="/customer/:id">
  {() => <ProtectedRoute component={() => <CustomerDetail />} />}
</Route>

// NACHHER (korrekt):
<Route path="/customer/:id">
  <ProtectedRoute component={CustomerDetail} />
</Route>
Gleiche Korrektur für alle betroffenen Routen:

Zeile 115: /customer/:id → component={CustomerDetail}
Zeile 133: /service-records/:id → component={ServiceRecordDetailPage}
Zeile 136: /appointment/:id → component={AppointmentDetail}
Zeile 142: /edit-appointment/:id → component={EditAppointment}
Zeile 145: /document-appointment/:id → component={DocumentAppointment}
Zeile 163: /admin/customers/:id/edit → component={AdminCustomerEdit}
Zeile 166: /admin/customers/:id → component={AdminCustomerDetail}


A5. MITTEL: Vite Build-Konfiguration nicht optimiert
Datei: vite.config.ts
Problem: Keine manualChunks-Konfiguration, kein Bundle-Splitting nach Vendor-Bibliotheken. Alle Dependencies landen in einem einzigen Chunk, was die initiale Ladezeit erhöht.
Anweisung:
typescript// vite.config.ts, build-Sektion erweitern:
build: {
  outDir: path.resolve(import.meta.dirname, "dist/public"),
  emptyOutDir: true,
  rollupOptions: {
    output: {
      manualChunks: {
        vendor: ['react', 'react-dom'],
        query: ['@tanstack/react-query'],
        ui: ['@radix-ui/react-dialog', '@radix-ui/react-select', '@radix-ui/react-tabs'],
        icons: ['lucide-react'],
        dates: ['date-fns'],
      },
    },
  },
},
Zusätzlich: Installiere rollup-plugin-visualizer um die Bundle-Größe zu analysieren:
bashnpm install -D rollup-plugin-visualizer
typescript// In vite.config.ts plugins:
import { visualizer } from 'rollup-plugin-visualizer';
// plugins: [..., visualizer({ open: true, gzipSize: true })]

A6. MITTEL: React Compiler nicht aktiviert
Problem: Die App nutzt React 19.2, aber der neue React Compiler (v1.0, seit Oktober 2025) ist nicht aktiviert. Dieser würde automatisch useMemo, useCallback und React.memo für alle Komponenten einfügen und laut Meta-Benchmarks 20-30% weniger Re-Renders erzeugen.
Anweisung:
bashnpm install --save-dev --save-exact babel-plugin-react-compiler@latest
typescript// vite.config.ts
export default defineConfig({
  plugins: [
    react({
      babel: {
        plugins: [
          ["babel-plugin-react-compiler", {}],
        ],
      },
    }),
    // ... andere Plugins
  ],
});
Hinweis: Nach Aktivierung des Compilers können die meisten manuellen useMemo/useCallback-Aufrufe entfernt werden. Der Compiler übernimmt das automatisch. Erst den Compiler aktivieren, dann testen, dann manuelle Memoization schrittweise entfernen.

A7. NIEDRIG: Fehlende Error Boundaries für Seiten-Gruppen
Datei: client/src/App.tsx
Problem: Es gibt eine globale ErrorBoundary (Zeile 199), aber keine seiten-spezifischen Error Boundaries. Wenn ein Fehler in der Admin-Kundenliste auftritt, wird die gesamte App mit einer Fehlerseite ersetzt — statt nur den betroffenen Bereich.
Anweisung:

Füge Error Boundaries um logische Seitengruppen hinzu:

tsxfunction AdminRoute({ component: Component }: { component: React.ComponentType }) {
  // ...
  return (
    <ErrorBoundary fallback={<AdminErrorFallback />}>
      <Suspense fallback={<PageLoader />}>
        <Component />
      </Suspense>
    </ErrorBoundary>
  );
}

### B6. HOCH: God-File `storage.ts` mit 1017 Zeilen

**Datei:** `server/storage.ts` (1017 Zeilen) + `server/storage/customer-management.ts` (559 Zeilen)

**Problem:** `storage.ts` enthält Methoden für Kunden, Termine, Service-Records, Appointment-Services und Systemeinstellungen — alles in einer Klasse. Das macht Wartung und Testing schwierig.

**Anweisung:**

B8. MITTEL: Kein Rate-Limiting
Problem: Keine Rate-Limiting-Middleware. Ein Bot oder fehlerhaftes Frontend könnte unbegrenzt Requests senden.
Anweisung:

import rateLimit from 'express-rate-limit';

const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 Minuten
  max: 500,                  // Max 500 Requests pro Fenster
  message: { error: "Zu viele Anfragen, bitte warten Sie." },
});

app.use('/api/', apiLimiter);

// Strengeres Limit für Auth-Routen:
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 20,
});
app.use('/api/auth/', authLimiter);
