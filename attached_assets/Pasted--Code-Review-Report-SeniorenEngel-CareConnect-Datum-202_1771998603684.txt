## Kritische Findings
 
Die folgenden Findings erfordern sofortige Bearbeitung, da sie Sicherheitsluecken, Datenintegritaet oder Patientensicherheit betreffen.
 
### CRIT-01: Fehlende Datenbank-Transaktionen bei Multi-Table-Operationen
 
- **Gefunden von:** Datenbank-Experte, Backend-Architekt
- **Dateien:**
  - `server/storage/customer-management.ts` Zeilen 270-384 (`createFullCustomer`)
  - `server/storage/budget-ledger.ts` Zeilen 1135-1293 (`createCascadeConsumption`)
  - `server/storage.ts` Zeilen 543-564 (`createErstberatungWithCustomer`)
- **Problem:** Multi-Table-Operationen (6+ Tabellen bei Kundenerstellung, Budget-Kaskade) werden OHNE `db.transaction()` ausgefuehrt. Bei Fehler mitten in der Operation bleiben inkonsistente Daten zurueck. Der manuelle Rollback in `createFullCustomer` (Zeile 380) ist nicht atomar -- wenn der Cleanup fehlschlaegt, bleiben Waisen-Datensaetze.
- **Auswirkung:** Datenkorruption bei Finanzdaten (Budget), verwaiste Kundendaten bei Erstellungsfehlern.
- **Loesung:**
  ```typescript
  async createFullCustomer(data: CreateFullCustomer, userId: number): Promise<Customer> {
    return db.transaction(async (tx) => {
      const [customer] = await tx.insert(customers).values(customerData).returning();
      await tx.insert(customerInsuranceHistory).values({...});
      await tx.insert(customerContacts).values({...});
      // alle weiteren Inserts mit tx statt db
      return customer;
    });
  }
  ```
  Fuer Budget-Operationen `isolationLevel: 'serializable'` verwenden.
 
### CRIT-02: SHA-256 Legacy-Passwort-Fallback
 
- **Gefunden von:** Security-Spezialist
- **Datei:** `server/services/auth.ts` Zeilen 27-36
- **Problem:** Die `verifyPassword`-Funktion akzeptiert noch Passwoerter im Format `salt:sha256hash`. SHA-256 ohne Key-Stretching ist mit modernen GPUs in Minuten knackbar. Eine Datenbank-Kompromittierung wuerde diese Passwoerter sofort offenlegen.
- **Loesung:**
  ```typescript
  // Bei erfolgreichem SHA-256 Login: sofort auf bcrypt upgraden
  if (hash === inputHash) {
    const bcryptHash = await hashPassword(password);
    await db.update(users).set({ passwordHash: bcryptHash }).where(eq(users.id, user.id));
    console.warn(`[SECURITY] Upgraded legacy SHA-256 hash for user ${user.id}`);
    return true;
  }
  ```
  Nach einer Grace Period den SHA-256-Pfad komplett entfernen und betroffene Accounts zum Passwort-Reset zwingen.
 
### CRIT-03: Keine Kunden-Anonymisierung (DSGVO Art. 17)
 
- **Gefunden von:** Security-Spezialist
- **Datei:** Fehlendes Feature -- Mitarbeiter-Anonymisierung existiert in `server/routes/admin/employees.ts` Zeilen 351-445, aber kein Aequivalent fuer Kunden.
- **Problem:** Kundendaten (Name, Adresse, Telefon, Geburtsdatum, Pflegegrad, Vorerkrankungen, Anamnese, Notfallkontakte, Versicherungsdaten) bleiben fuer immer in der DB. Unter DSGVO Art. 17 haben Betroffene ein Recht auf Loeschung. Unter Art. 5(1)(e) duerfen Daten nicht laenger als noetig aufbewahrt werden.
- **Loesung:** `anonymizeCustomer()`-Funktion analog zu `anonymizeEmployee()` implementieren, die PII in allen verknuepften Tabellen ersetzt (customers, customer_contacts, customer_needs_assessments, customer_care_level_history, customer_pricing_history, customer_insurance_history). Terminaufzeichnungen mit anonymisierter Kundenreferenz fuer Abrechnungszwecke beibehalten.
 
### CRIT-04: Notfall-/Gesundheitsinfos nicht auf Termin-Ansicht sichtbar
 
- **Gefunden von:** UX-Experte
- **Dateien:**
  - `client/src/features/appointments/components/appointment-card.tsx`
  - `client/src/pages/appointment-detail.tsx`
- **Problem:** Pflegekraefte sehen auf der Terminseite NICHT: Allergien/Vorerkrankungen, Notfallkontakte, Haustierinformationen, Pflegegrad. Diese Daten existieren im System, sind aber nur auf der Admin-Kundendetailseite erreichbar. Fuer eine Pflegekraft, die beim Patienten ankommt, ist das ein Sicherheitsrisiko.
- **Loesung:** "Wichtige Hinweise"-Sektion auf der Termin-Detailseite hinzufuegen: Pflegegrad-Badge, Vorerkrankungen (mit Warn-Icon), Haustierinformationen, primaerer Notfallkontakt mit Click-to-Call.
 
### CRIT-05: Kein Rate Limiting
 
- **Gefunden von:** Backend-Architekt, Security-Spezialist
- **Datei:** `server/index.ts` (komplett fehlend)
- **Problem:** Kein einziger Endpoint hat Rate Limiting. Login, Passwort-Reset, API-Endpunkte sind ungeschuetzt gegen Brute-Force, Credential Stuffing, und DoS.
- **Loesung:**
  ```typescript
  import rateLimit from "express-rate-limit";
  // Global: 200 Requests/min pro IP
  app.use("/api", rateLimit({ windowMs: 60000, max: 200, standardHeaders: true }));
  // Login: 5 Versuche/15 min pro IP
  router.post("/login", rateLimit({ windowMs: 15 * 60 * 1000, max: 5 }), ...);
  // Passwort-Reset: 3 Versuche/Stunde pro IP
  router.post("/password-reset/request", rateLimit({ windowMs: 3600000, max: 3 }), ...);
  ```

### CRIT-07: Dualer API-Client mit 50+ rohen fetch()-Calls
 
- **Gefunden von:** Frontend-Architekt
- **Dateien:**
  - Legacy: `client/src/lib/queryClient.ts` Zeilen 3-31, 40-67
  - Neu: `client/src/lib/api/client.ts` Zeilen 10-42, 138-257
  - 50+ rohe `fetch()` Calls verteilt ueber: `pages/service-records.tsx`, `pages/service-record-detail.tsx`, `pages/tasks.tsx`, `pages/admin/components/customer-documents-section.tsx`, `pages/admin/components/employee-documents-section.tsx`, `features/appointments/hooks/use-appointments.ts`, `features/tasks/use-tasks.ts`, `components/layout.tsx`, `pages/customer-detail.tsx`, und 20+ weitere
- **Problem:** Zwei separate API-Abstraktionen mit unterschiedlicher CSRF-Cookie-Parsing-Logik. Der Legacy-Client parsed Cookies mit `.split("=")` (fehlerhaft bei Cookies mit `=` im Wert), der neue mit `indexOf("=")` (korrekt). 50+ Stellen umgehen beide Clients komplett -- ohne Retry-Logik, mit inkonsistenter Fehlerbehandlung, teilweise ohne `credentials: "include"`.
- **Loesung:** Legacy-Funktionen aus `queryClient.ts` entfernen (nur QueryClient-Instanz behalten). Alle fetch()-Calls systematisch auf `api.get/post/patch/delete` aus `lib/api/client.ts` migrieren. Fuer React Query `createQueryFn` verwenden.

## Security & DSGVO
 
### SEC-01: Fehlende HTTP Security Headers [HIGH]
 
- **Datei:** `server/index.ts` (fehlend)
- **Problem:** Kein CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy.
- **Loesung:** `npm install helmet` und konfigurieren:
  ```typescript
  import helmet from "helmet";
  app.use(helmet({
    contentSecurityPolicy: {
      directives: {
        defaultSrc: ["'self'"],
        scriptSrc: ["'self'"],
        styleSrc: ["'self'", "'unsafe-inline'"],
        imgSrc: ["'self'", "data:", "https://storage.googleapis.com"],
      },
    },
    hsts: { maxAge: 31536000, includeSubDomains: true, preload: true },
  }));
  ```
 
### SEC-02: CSRF-Schutz fehlt auf Auth-Routes [HIGH]
 
- **Datei:** `server/routes/index.ts` Zeilen 26-28
- **Problem:** Auth-Router wird VOR dem CSRF-Middleware gemountet. `/api/auth/logout`, `/api/auth/password-reset/request`, `/api/auth/password-reset/confirm`, `/api/auth/setup` haben keinen CSRF-Schutz.
- **Loesung:** Auth-Router nach CSRF-Middleware mounten, oder CSRF individuell auf state-changing Auth-Endpoints anwenden (ausser `/auth/login`).
 
### SEC-03: Keine Account-Lockout-Mechanik [HIGH]
 
- **Datei:** `server/services/auth.ts` Zeilen 113-128
- **Problem:** Unbegrenzte Login-Versuche ohne Counter, Lockout oder progressive Verzoegerung.
- **Loesung:** `failedLoginAttempts` und `lockedUntil` Spalten in `users`-Tabelle. Nach 5 Fehlversuchen 15 min Sperre. Reset bei erfolgreichem Login.
 
### SEC-04: .gitignore fehlt .env-Schutz [HIGH]
 
- **Datei:** `.gitignore`
- **Problem:** Keine `.env`, `.env.*`, `*.pem`, `*.key` Eintraege. Ein versehentliches `git add .` wuerde Secrets committen.
- **Loesung:** Sofort ergaenzen:
  ```
  .env
  .env.*
  .env.local
  .env.production
  .env.development
  *.pem
  *.key
  ```
 
### SEC-05: Unvollstaendiger Audit-Trail [HIGH]
 
- **Datei:** `shared/schema/audit.ts` Zeilen 6-17
- **Problem:** Audit deckt nur Termine und Leistungsnachweise ab. Fehlend: Login/Logout, fehlgeschlagene Logins, Passwortwechsel, Kundendaten-CRUD, Zugriff auf Gesundheitsdaten, Rollenänderungen, Admin-Aktionen, Systemeinstellungen.
- **Loesung:** `AUDIT_ACTIONS` erweitern um mindestens: `login_success`, `login_failed`, `password_changed`, `customer_created`, `customer_updated`, `customer_viewed`, `customer_anonymized`, `role_changed`, `settings_updated`.
 
### SEC-06: Keine Datenaufbewahrungsfristen [HIGH]
 
- **Datei:** `server/services/auth.ts` Zeilen 474-490; `server/index.ts` Zeilen 62-70
- **Problem:** `cleanupExpiredSessions()` und `cleanupExpiredResetTokens()` existieren, werden aber nie aufgerufen. Kein Scheduler fuer Cleanup. Sessions und Reset-Tokens akkumulieren unbegrenzt.
- **Loesung:** Cleanup in `setInterval` einbinden (stuendlich).
 
### SEC-07: Kein Consent-Management [HIGH]
 
- **Problem:** Kein System zur Einwilligungserfassung fuer Gesundheitsdaten (DSGVO Art. 9). Keine Aufzeichnung wann/wozu Einwilligung erteilt wurde.
- **Loesung:** `customer_consents`-Tabelle mit: Einwilligungstyp, Datum, Rechtsgrundlage, Umfang, Widerrufsereignisse. Integration in Erstberatungs-Workflow.
 
### SEC-08: Keine Verschluesselung fuer Gesundheitsdaten [MEDIUM]
 
- **Datei:** `server/lib/db.ts` Zeilen 7-14
- **Problem:** DB-Verbindung erzwingt kein TLS. Gesundheitsdaten (Pflegegrad, Vorerkrankungen, Anamnese) liegen unverschluesselt in der DB.
- **Loesung:** TLS fuer DB-Verbindung erzwingen. Langfristig: Column-Level Encryption fuer medizinische Felder.
 
### SEC-09: Session ohne Rotation [MEDIUM]
 
- **Datei:** `server/services/auth.ts` Zeilen 113-146; `server/middleware/auth.ts` Zeilen 27-35
- **Problem:** Session-Token wird bei Login erstellt und fuer 7 Tage nie rotiert. Kein Sliding Expiry.
- **Loesung:** Sliding Expiry + periodische Token-Rotation.
 
### SEC-10: Schwache Passwort-Policy [MEDIUM]
 
- **Datei:** `shared/schema/users.ts` Zeile 108; `server/routes/auth.ts` Zeilen 215-223
- **Problem:** Minimum nur 8 Zeichen, keine Komplexitaetsanforderungen, Passwortaenderung benoetigt kein aktuelles Passwort.
- **Loesung:** Minimum 12 Zeichen, aktuelles Passwort bei Aenderung verlangen, Breached-Password-Check.
 
### SEC-11: Passwort-Reset nicht funktional [MEDIUM]
 
- **Datei:** `server/routes/auth.ts` Zeilen 152-178
- **Problem:** Token wird erstellt und gehasht gespeichert, aber nie per E-Mail versandt. Kommentar sagt "In Produktion wurde hier eine E-Mail gesendet".
- **Loesung:** Transactional E-Mail Service integrieren (SES, Mailgun). Alte Tokens bei neuem Request invalidieren.
 
### SEC-12: CSRF-Token nicht an Session gebunden [MEDIUM]
 
- **Datei:** `server/middleware/csrf.ts` Zeilen 21-55
- **Problem:** CSRF-Token ist ein standalone Random Value, nicht kryptographisch an die Session gebunden. Bei Cookie-Injection (Subdomain XSS) umgehbar.
- **Loesung:** HMAC-Signatur mit Server-Secret + Session-Token-Hash.
 
### SEC-13: Object Storage ohne Customer-Level Auth [MEDIUM]
 
- **Datei:** `server/replit_integrations/object_storage/routes.ts` Zeilen 33-44
- **Problem:** Download-Route prueft Authentifizierung, aber nicht ob der User auf das spezifische Objekt zugreifen darf. `canAccessObjectEntity` existiert, wird aber nicht aufgerufen.
- **Loesung:** Authorization-Check vor Download einfuegen.
 
### SEC-14: Template Engine XSS [MEDIUM]
 
- **Datei:** `server/services/template-engine.ts` Zeilen 122-130
- **Problem:** Kundendaten werden per Regex in HTML-Templates eingefuegt OHNE HTML-Escaping. `escapeHtml`-Funktion existiert (Zeile 150), wird aber nur fuer Dokumenttitel verwendet.
- **Loesung:** `escapeHtml()` auf alle Placeholder-Werte anwenden.
 
### SEC-15: Setup-Endpoint Race Condition [MEDIUM]
 
- **Datei:** `server/routes/auth.ts` Zeilen 248-301
- **Problem:** `/auth/setup` prueft `hasAnyAdmin()`, dann erstellt Admin. TOCTOU: Zwei gleichzeitige Requests koennten beide die Pruefung passieren.
- **Loesung:** Serializable Transaction mit Check + Insert.
 
### SEC-16: JSON Body ohne Size Limit [MEDIUM]
 
- **Datei:** `server/index.ts` Zeile 13
- **Problem:** `express.json()` ohne `limit`. Oversized Payloads (Base64 Signaturen) koennen Server-Memory verbrauchen.
- **Loesung:** `express.json({ limit: "2mb" })`.
 
### SEC-17: Signatur-Daten ohne Max-Laenge [MEDIUM]
 
- **Dateien:** `shared/schema/service-records.ts` Zeile 69; `shared/schema/appointments.ts` Zeile 39; `shared/schema/documents.ts` Zeilen 87-88
- **Problem:** Signature-Data (Base64 PNG) hat nur `min(1)` Validierung, kein Maximum. Beliebig grosse Strings moeglich.
- **Loesung:** `.max(500000)` und Regex-Validierung fuer Base64-Image-Format.
 
### SEC-18: Admin Self-Protection mit unvalidiertem Input [MEDIUM]
 
- **Datei:** `server/routes/admin/employees.ts` Zeilen 141-155
- **Problem:** Self-Protection-Checks nutzen `req.body` statt Zod-validierte Daten. Kein "letzter Admin"-Schutz.
- **Loesung:** Checks nach Zod-Validierung verschieben. `count()` auf aktive Admins bevor `isAdmin: false` erlaubt wird.
 
### SEC-19: Signatur-Integritaet ohne HMAC [LOW]
 
- **Datei:** `server/services/signature-integrity.ts` Zeilen 1-5
- **Problem:** Plain SHA-256 statt HMAC. Bei DB-Zugriff koennten Signatur und Hash gleichzeitig manipuliert werden.
- **Loesung:** HMAC-SHA-256 mit Server-seitigem Secret.
 
### SEC-20: Batch-Services ohne Customer-Access-Check [LOW]
 
- **Datei:** `server/routes/appointments.ts` Zeilen 110-123
- **Problem:** `GET /appointments/batch-services` prueft nicht ob der User auf die Kunden der Termine zugreifen darf.
- **Loesung:** Ergebnisse durch zugewiesene Kunden-IDs filtern.
 
### SEC-21: Session Cookie sameSite=lax statt strict [LOW]
 
- **Datei:** `server/middleware/auth.ts` Zeilen 27-35
- **Problem:** Session-Cookie nutzt `sameSite: "lax"`, CSRF-Cookie nutzt `"strict"` -- Inkonsistenz.
- **Loesung:** Session-Cookie auf `sameSite: "strict"` aendern.

## Backend & API
 
### BE-01: Fat Route Handlers mit Business-Logik [HIGH]
 
- **Dateien:**

### BE-07: Passwortaenderung invalidiert nicht andere Sessions [MEDIUM]

### BE-08: Environment Variables ohne Validierung [MEDIUM]

### BE-09: `any`-Types im Storage Interface [MEDIUM]
 
- **Problem:** Einige Storage-Methoden nutzen `any` als Rueckgabetyp statt typisierter Interfaces.
- **Loesung:** Alle `any`-Types durch konkrete Typen ersetzen.
 
### BE-10: Error-Matching per String [MEDIUM]
 
- **Problem:** Error-Handler vergleichen Fehlermeldungen als Strings statt Error-Klassen zu nutzen.
- **Loesung:** Dedizierte Error-Klassen (`NotFoundError`, `ValidationError`, etc.) konsequent nutzen.
 
### BE-11: Tote Code in Error-Modul [LOW]

## Datenbank & Datenintegritaet
 
### DB-01: Fehlende CHECK Constraints auf Enum-Spalten [HIGH]
 
- **Dateien:** Nahezu alle Schema-Dateien mit `text()`-Spalten fuer Enums (status, appointmentType, role, entryType, billingType, contactType, budgetType, transactionType, etc.)
- **Problem:** Keine DB-Level-Erzwingung fuer gueltige Werte. Nur Zod validiert, aber direkter DB-Zugriff, Migration-Scripts oder ORM-Bugs koennen ungueltige Werte einfuegen.
- **Loesung:** PostgreSQL CHECK Constraints oder `pgEnum` fuer alle Enum-Spalten:
  ```sql
  ALTER TABLE appointments ADD CONSTRAINT appointments_status_check
    CHECK (status IN ('scheduled', 'in-progress', 'documenting', 'completed', 'cancelled'));
  ```
 
### DB-02: Fehlende NOT NULL Constraints auf kritischen Feldern [HIGH]
 
- **Datei:** `shared/schema/customers.ts` Zeilen 16-27
- **Problem:** `vorname`, `nachname`, `strasse`, `nr`, `plz`, `stadt` sind nullable, obwohl sie fuer aktive Kunden erforderlich sein sollten (Patientenidentifikation).
- **Loesung:** NOT NULL Constraints mit Backfill-Migration, oder CHECK Constraint konditioniert auf Status.
 
### DB-03: Fehlender Soft-Delete-Index [HIGH]
 
- **Datei:** `shared/schema/appointments.ts` Zeilen 45-53
- **Problem:** Fast jede Appointment-Query filtert auf `isNull(appointments.deletedAt)`, aber kein Index deckt `deletedAt` ab. Geloeschte Zeilen werden bei jeder Abfrage durchgescannt.
- **Loesung:** Partial Index: `CREATE INDEX appointments_active_idx ON appointments (date) WHERE deleted_at IS NULL;`
 
### DB-04: N+1 Query in `createFullCustomer` [HIGH]
 
- **Datei:** `server/storage/customer-management.ts` Zeilen 270-384
- **Problem:** 8+ sequentielle DB-Queries ohne Transaction. Jeder ist ein separater Round-Trip zur Neon Serverless DB.
- **Loesung:** Alles in einer `db.transaction()` buendeln (loest gleichzeitig CRIT-01).
 
### DB-05: Budget-Summary holt alle Zeilen client-seitig [HIGH]
 
- **Datei:** `server/storage/budget-ledger.ts` Zeilen 325-402
- **Problem:** `getBudgetSummary` holt ALLE Allocations und Transactions fuer einen Kunden, aggregiert dann in JavaScript mit `.reduce()`. Bei Langzeit-Kunden tausende Zeilen.
- **Loesung:** SQL-Aggregation statt JavaScript:
  ```sql
  SELECT COALESCE(SUM(amount_cents), 0) as total_allocated
  FROM budget_allocations WHERE customer_id = $1 AND budget_type = $2
    AND valid_from <= $3 AND (expires_at IS NULL OR expires_at >= $3);
  ```

Die folgenden Aspekte des Codes sind gut umgesetzt und sollten beibehalten werden:
 
### Security
- Bcrypt mit 12 Rounds fuer Passwort-Hashing

- Zod-Validierung auf fast allen Endpoints

- 29/32 Routes lazy-loaded mit React.lazy() und Suspense
- Design-System Token Architektur (tokens.ts) -- gut entworfen, Adoption muss steigen
