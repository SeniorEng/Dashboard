markdown# CareConnect — Anweisungsdokument für Replit Agent

## Überblick

Dieses Dokument enthält priorisierte, ausführbare Anweisungen aus einem vollständigen Code-Audit.
Arbeite die Issues **von oben nach unten** ab. Nach jeder Änderung:

1. `npm run check` — TypeScript-Fehler prüfen
2. `npm run build` — Produktions-Build testen
3. App im Browser testen (betroffene Funktion manuell durchklicken)

### 1.1 XSS: Template-HTML wird ohne Sanitisierung gerendert

**Dateien:**
- `client/src/pages/admin/document-templates.tsx` (Zeile ~544: `dangerouslySetInnerHTML`)
- `client/src/pages/admin/components/signatures-step.tsx` (Zeilen 124-127: `window.open()` + `document.write()`)
- `server/services/template-engine.ts` (`renderTemplate()`)

**Problem:** Admin-erstellter HTML-Content UND Kundendaten (Name, Adresse) werden direkt in HTML eingefügt ohne Escaping/Sanitisierung. Ein Kundenname wie `<script>alert('xss')</script>` wird ausgeführt. Die Funktion `escapeHtml()` existiert bereits im Code, wird aber nur für den `<title>`-Tag verwendet.

**Anweisung:**

Schritt 1 — Dependency installieren:```bash
npm install dompurify @types/dompurify```

Schritt 2 — Client-seitig in `document-templates.tsx` und `signatures-step.tsx`:
Überall wo `dangerouslySetInnerHTML` oder `document.write()` mit DB-Inhalt genutzt wird, vorher sanitisieren:```typescript
import DOMPurify from "dompurify";
const safeHtml = DOMPurify.sanitize(previewHtml, {
  ALLOWED_TAGS: ['h1','h2','h3','p','br','strong','em','ul','ol','li','table','tr','td','th','thead','tbody','img','div','span'],
  ALLOWED_ATTR: ['class','style','src','alt','width','height','colspan','rowspan'],
});```

Schritt 3 — Server-seitig in `server/services/template-engine.ts` → `renderTemplate()`:
Escape ALLE Placeholder-Werte (außer Signatur-`<img>`-Tags):```typescript
for (const [key, value] of Object.entries(placeholders)) {
  const safeValue = key.includes("signature") ? value : escapeHtml(value);
  result = result.replace(new RegExp(`\\{\\{${key}\\}\\}`, "g"), safeValue);
}```

**Test:** Template mit `<script>alert('test')</script>` als Kundenname erstellen → darf NICHT als Script ausgeführt werden.

const errors: { slug: string; error: string }[] = [];
for (const sig of parsed.data.signatures) {

  return res.status(500).json({ error: "Alle Unterschriften fehlgeschlagen", details: errors });
}
res.status(results.length === parsed.data.signatures.length ? 201 : 207).json({ results, errors });```

**Anweisung Client** (`client/src/pages/admin/customer-new.tsx`, Zeilen 246-268):```typescript

### 1.4 Kundenanlage ohne Datenbank-Transaktion

**Datei:** `server/routes/admin/customers.ts` Zeilen 227-357

**Problem:** POST `/customers` führt 6+ DB-Operationen sequentiell aus (Kunde → Pflegegrad → Versicherung → Kontakte → Budgets → Vertrag), aber ohne Transaktion. Wenn Schritt 4 fehlschlägt, sind Schritte 1-3 committed → inkonsistente Daten.

**Anweisung:** Alle Operationen in eine Transaktion wrappen:```typescript
const result = await db.transaction(async (tx) => {
  const customer = await tx.insert(customers).values(customerData).returning();
  if (data.pflegegrad) await addCareLevelHistory({...}, userId, tx);
  if (data.insurance) await addCustomerInsurance({...}, userId, tx);
  if (data.contacts) await tx.insert(customerContacts).values(contactsArray);
  // ... alle Operationen innerhalb der Transaktion
  return customer[0];
});
res.status(201).json(result);```

**Wichtig:** Alle Storage-Funktionen (z.B. `addCareLevelHistory`, `addCustomerInsurance`, `addCustomerContact`) müssen ein optionales `tx`-Parameter akzeptieren und statt `db` dann `tx` nutzen. Prüfe auch `server/storage/customer-management.ts` `createFullCustomer()` (Zeilen 270-384) auf das gleiche Problem.

**Test:** Kunden mit absichtlich fehlerhaften Kontaktdaten anlegen → entweder alles gespeichert oder nichts.

### 1.5 `staleTime: Infinity` — Daten werden nie automatisch aktualisiert

**Datei:** `client/src/lib/queryClient.ts` Zeilen 87-100

**Problem:** Globaler Default `staleTime: Infinity` + `refetchOnWindowFocus: false` + `retry: false`. Wenn Mitarbeiter A einen Termin dokumentiert, sieht Mitarbeiter B die Änderung erst nach komplettem Seiten-Reload.

**Anweisung:**```typescript
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      queryFn: getQueryFn({ on401: "throw" }),
      refetchInterval: false,
      refetchOnWindowFocus: true,        // WAR: false
      staleTime: 2 * 60 * 1000,         // WAR: Infinity → 2 Minuten
      gcTime: 10 * 60 * 1000,           // NEU: 10 Min Garbage Collection
      retry: 1,                          // WAR: false → 1 Retry
    },
    mutations: {
      retry: false,
    },
  },
});```

**Test:** Öffne App in 2 Tabs → ändere Termin in Tab 1 → wechsle zu Tab 2 → nach wenigen Sekunden muss Änderung sichtbar sein.

### 1.6 Route-Pattern erzeugt neue Komponenten bei jedem Render

**Datei:** `client/src/App.tsx` Zeilen 115, 133, 136, 142, 145, 163, 166

**Problem:**```tsx
// SCHLECHT:

  {() => <ProtectedRoute component={() => } />}
```
Das doppelt verschachtelte `() =>` erzeugt bei jedem Render 2 neue anonyme Funktionen. React sieht eine "neue" Komponente → unmountet die alte → verliert State → startet alle Queries neu.

**Anweisung — alle 7 betroffenen Routen korrigieren:**```tsx
// GUT:

  
```

Betroffene Routen:
- Zeile 115: `/customer/:id` → `component={CustomerDetail}`
- Zeile 133: `/service-records/:id` → `component={ServiceRecordDetailPage}`
- Zeile 136: `/appointment/:id` → `component={AppointmentDetail}`
- Zeile 142: `/edit-appointment/:id` → `component={EditAppointment}`
- Zeile 145: `/document-appointment/:id` → `component={DocumentAppointment}`
- Zeile 163: `/admin/customers/:id/edit` → `component={AdminCustomerEdit}`
- Zeile 166: `/admin/customers/:id` → `component={AdminCustomerDetail}`

**Test:** Navigiere zu Kundendetail → zurück → wieder hin → State (z.B. Tab-Auswahl) muss erhalten bleiben.
